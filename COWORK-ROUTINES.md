# Cowork / Claude Code 定期タスク仕様（情報源の発掘・審査・順位付け + 嗜好レポート）

このファイルは、デイリーブリーフの「判断が要る・低頻度・クリティカルパス外」の
仕事を、週1の定期タスク(Cowork または Claude Code の Routine)に任せるための
**仕様書 兼 プロンプト全文**です。ユーザーはこの中の「週次Routineプロンプト」を
自分の定期タスクに貼って設置します。SYSTEM-DESIGN §7・§8 の設計をそのまま実装
したもの。

---

## どこで動かすか

- **コワーク / コード どちらでも動く**(両方 Anthropic のクラウドで定期実行が可能)。
- 推奨: **Claude Code の定期タスク**。理由は my-brain が git リポジトリで、
  clone→編集→commit→push がネイティブで確実、Supabase も Bash の curl で読める
  (コネクタ設定不要)。コワークで動かす場合は GitHub コネクタの接続が要る。
- スケジュール: **週1・オフ時間(例: 日曜 03:00 JST)**。クリティカルパス外。
  走らない週があっても、夜間Cron(Gemini)は現状の sources.md をそのまま巡回し
  続けるので、ブリーフは止まらない(graceful degradation)。

### ★Claude Code(コード)で動かす時の必須設定2つ(2026-07-20 実機で判明)

1. **環境のネットワークアクセスを「Full」にする**。ルーティン編集画面の環境
   セレクタ(☁)→「環境を追加」→**ネットワークアクセス=Full**(無制限)で環境を
   作り、その環境をこのルーティンに割り当てる。既定の「デフォルト」環境は
   **Trusted**(npm等パッケージ取得のみ許可・一般Webは403でブロック)なので、
   Trustedのままだと WebFetch/curl が全ホスト403になり発掘できない
   (`example.com`すら弾かれる=サイト側でなく出口ポリシーの403。curl小細工でも
   越えられない)。Full は「なし/Trusted/Full/カスタム」の選択肢の一つ。
2. **デフォルトブランチへ直接コミットさせる**(プロンプトの「# コミット」節)。
   コードは既定で新しい `claude/...` ブランチへpushしPR待ちで止まる=デフォルト
   ブランチに入らずアプリに反映されない。プロンプトで直コミットを明示する。
   それでもブランチが作られたら、GitHubで「Compare & pull request」→Merge で
   取り込む(または設定→Claude Code「プルリクエストを自動的に作成する」+GitHub
   auto-merge)。

## このタスクが持つ責務

日次のブリーフ生成(巡回→抽出→分類→編成)はアプリ側 Gemini(Vercel Cron)が
毎日やる。このタスクはそこに**乗せる情報源の質と並び**、および**嗜好の
振り返り**だけを持つ。

1. **情報源の発掘・審査**(§3-4/§3-7): 好み・興味に合う質の高いサイトを探し、
   審査し、合格分を情報源プールへ足す。
2. **情報源プールの順位付け・淘汰**(§7-4): logs/ の反応で情報源を打率順に並べ替え、
   上限(〜60)を超えたら実績の低いものを落とす。固定(アプリ内蔵FIXED_SOURCES)・
   お気に入り(ユーザー登録)は淘汰しない。
3. **嗜好の差分レポート**(§7-6/§8): KEEP/SKIPログを分析し、好み・語り口・
   深さの傾向と変化を人間可読レポートにして書き戻す(Obsidianで読む)。
   taste-state.md の好み・興味は書き換えない(確定はユーザー本人)。

## 必要なアクセス・環境変数(タスク側に設定)

- my-brain リポジトリの git 読み書き(本人のリポジトリ)。
- Web 検索・閲覧(発掘・審査用)。
- Supabase(**読み取りのみ**): `SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` /
  `OWNER_USER_ID`。

## 扱うファイル(my-brain)

- `taste-state.md`(**読むだけ**): 「## 好み」「## 興味」「## 生活圏」「## 願い」。
- `sources.md`: 情報源リスト。**アプリ管理の
  `<!-- BEGIN app-managed:favorites -->`〜`<!-- END app-managed:favorites -->`
  ブロックには絶対に触れない**(お気に入り=ユーザー登録・淘汰対象外)。
  このタスクが管理するのは下記 Cowork ブロックだけ:
  ```
  <!-- BEGIN cowork:discovered -->
  ## 発掘した情報源（KEEP実績順）
  - [サイト名](https://example.com) — 一行の採用理由 ｜ 実績: KEEP 3 / SKIP 1
  <!-- END cowork:discovered -->
  ```
  アプリの夜間Cronは sources.md 内の全箇条書きURLを読むので、このブロックの
  URLも自動で巡回対象になる(アプリ改修不要)。
- `cowork/rejected.md`: 審査で不合格にしたサイトのURLと理由(再審査回避)。
- `taste-report.md`: 嗜好の差分レポート(このタスクが毎回まるごと書き直す)。

## ★コワークにはGitHubコネクタが無い場合の半自動版（2026-07-20 実機で判明）

コワーク(コネクタ方式)には GitHub コネクタが用意されていないことがある
(実機で確認: Gmail/Calendar/Drive/Spotify/TickTick のみ、GitHub検索でも出ない)。
その場合コワークからは my-brain に直接書けない。**git書き込みを諦め、コワークは
「発掘して一覧で出す」だけにして、登録はアプリの「設定 → お気に入りの情報源」
から人手で行う**(この欄は元々 my-brain の sources.md へ同期され夜間生成に使われる)。
GitHub連携ゼロで今すぐ回せる。フル自動(git直書き)がしたい場合は、コネクタでなく
git ネイティブな **Claude Code(コード)側でタスクを作り、my-brain をリポジトリに
追加**する(下の「どこで動かすか」参照)。

### 半自動版プロンプト全文（コワーク・GitHub不要・出力を人手で登録）

```
# 役割
あなたはデイリーブリーフというアプリの情報源スカウトです。ユーザーが関心を持つテーマについて、質の高い書き手による記事を継続的に発信しているWebサイトを探し、記事を実際に読んで審査し、合格したサイトを一覧で出力します。ファイルへの書き込みはしません（ユーザーがアプリに手で登録します）。

# 前提（この下にユーザーが自分の値を書いておく）
好み: 建築, 器・工芸, 西洋美術
興味: 西洋美術の展覧会, ミニシアターの単館上映
生活圏: 東京23区
既に登録済みで重複させないサイト: （あれば列挙）

# 探す
「好み」「興味」の各テーマを検索語にして、それらを扱う記事・サイトをWeb検索する。見つけた良いサイトや記事が本文中でリンク・言及している別のサイトもたどる。特定のサイト名を検索語にして「それに似たサイト」を探すことはしない。

# 審査する
運営者が個人か組織かは合否に関係しない。サイトの規模や更新頻度でなく、文章そのもので判断する。候補サイトごとに、記事を2〜3本（できれば別々のテーマ）と、サイトの紹介ページ（"About"・「このサイトについて」・「運営者情報」など運営方針や書き手の立場が書かれた場所）を実際に開いて読む。
次の4つをすべて満たすサイトだけを合格とする:
1. 事実の紹介・要約だけで終わらず、書き手自身の評価・解釈・主張が含まれている。
2. プレスリリース・Wikipedia・他サイトの言い換えでない、その書き手独自の情報や観察が含まれている。
3. 固有名・年代・背景・経緯に具体的に踏み込んでおり、その分野に精通した人が書いたと判断できる情報量がある。
4. サイトの紹介ページ（"About"・「このサイトについて」・「運営者情報」など）に、運営方針・編集意図・書き手の関心や立場が述べられており、それが実際の記事の内容と一致している。紹介ページが見当たらないサイトは、記事全体から一貫した視点がはっきり読み取れる場合に限り合格とし、読み取れなければ不合格とする。
次のいずれかに当てはまるサイトは不合格とする: 他サイトからの寄せ集め・まとめ・ランキングが中心で書き手自身の文章が乏しい / アフィリエイトリンクや「おすすめ◯選」形式が主な内容 / 検索流入狙いのキーワードの繰り返しが目立つ / プレスリリースやニュースをほぼ転載している / 全文がペイウォール内・ログイン必須・本文がJavaScriptでしか表示されない。
判断に迷った場合は不合格とする。

# 出力
合格したサイトを最大5件、1件ずつ「サイト名 / URL / 合格と判断した具体的な理由（どの記事のどこがどう良かったか）」の形式で出す。続けて、不合格にした候補を「URL — 一行の不合格理由」で挙げる。

# 守ること
- 実際に開いて本文を読んだサイトだけを出す。URLを推測で作らない。
```

登録手順: アプリの **設定 → お気に入りの情報源** に、出力されたURLを1件ずつ貼って追加する(この欄が my-brain の sources.md へ同期され、夜間生成の巡回対象になる)。

## 段階を分けて設置する

初めて設置するときは、**まず「発掘だけのかんたん版」から始める**のが安全。
かんたん版は my-brain(GitHub)と Web 検索だけで動き、**Supabaseのキーは要らない**。
KEEP/SKIP の実績が数週間ぶん貯まってから、下の「フル版」に差し替えると、
順位付け・淘汰・嗜好レポートが加わる。

## かんたん版プロンプト全文（まずこれを設置する・Supabase不要）

```
# 役割
あなたはデイリーブリーフというアプリの情報源スカウトです。ユーザーが関心を持つテーマについて、質の高い書き手による記事を継続的に発信しているWebサイトを探し、記事を実際に読んで審査し、合格したサイトのURLだけを my-brain の sources.md に追記します。個々の記事を抜き出したりカードにしたりはしません。

# 最初に読む
存在しないファイルは飛ばす。
- taste-analysis.md: 「## 発掘の手がかり」「## これから好みそうな傾向」「## 安定した好み」。これが探す方向の主な手がかり（どんな書き手・サイト・切り口を、なぜ探すか）。
- taste-state.md の「## 生活圏」: 対象とする地域。
- sources.md の全体: すでに登録済みのURL。これらは提案しない。
- sources-user.md: アプリで削除された情報源(「## 除外」)。これらはプールに入れず、sources.md の cowork:discovered ブロックに在れば今回の書き換えで取り除く。
- cowork/rejected.md: 過去に不合格にしたURL。これらは再提案しない。
- logs/ の feedback-*.md（反応の記録。各行は「日付｜反応｜題名｜分野｜ドメイン｜要約」。反応は 残した/流した/拒否/実行/星付き）。cowork:discovered の各サイトの打率（下記「並べ替えと淘汰」）を出すのに使う。

# 探す
1. taste-analysis.md の「## 発掘の手がかり」「## これから好みそうな傾向」「## 安定した好み」を手がかりに、探すべき方向（テーマ・書き手・切り口）を決め、それらを扱う記事・サイトをWeb検索する。
2. 見つけた良いサイトや記事が、本文中でリンク・言及している別のサイトをたどる。
特定のサイト名を検索語にして「それに似たサイト」を探すことはしない。

# 審査する
運営者が個人か組織かは合否に関係しない。サイトの規模や更新頻度でなく、文章そのもので判断する。候補サイトごとに、次を実際に開いて読む:
- 記事を2〜3本（できれば別々のテーマのもの）
- サイトの紹介ページ（"About"・「このサイトについて」・「運営者情報」など。運営方針・書き手の立場・目的が書かれている場所）

読んだ内容が、次の4つをすべて満たすサイトだけを合格とする:
1. 事実の紹介・要約だけで終わらず、書き手自身の評価・解釈・主張が文章に含まれている。
2. プレスリリース・Wikipedia・他サイトの言い換えではない、その書き手独自の情報や観察が含まれている。
3. 固有名・年代・背景・経緯に具体的に踏み込んでおり、その分野に精通した人が書いたと判断できる情報量がある。
4. サイトの紹介ページ（"About"・「このサイトについて」・「運営者情報」など）に、運営方針・編集意図・書き手の関心や立場が述べられており、それが実際の記事の内容と一致している。紹介ページが見当たらないサイトは、記事全体から一貫した視点がはっきり読み取れる場合に限り合格とし、読み取れなければ不合格とする。

次のいずれかに当てはまるサイトは不合格とする:
- 他サイトからの寄せ集め・まとめ・ランキングが中心で、書き手自身の文章が乏しい。
- アフィリエイトリンクや「おすすめ◯選」形式の記事が主な内容。
- 検索流入を狙ったキーワードの繰り返しが目立つ。
- プレスリリースやニュースをほぼそのまま転載している。
- 全文がペイウォールの内側にある、ログインが必須、または本文がJavaScriptでしか表示されない。

判断に迷った場合は不合格とする。

# 並べ替えと淘汰（logs/ の反応データに基づく）
夜間の巡回は、この cowork:discovered の並びの「上位ほど少しだけ選ばれやすく、基本は上から下までランダム」に選ぶ。だから並び順＝打率順にしておくと、良いサイトが自然に多く使われ、悪いサイトは後退する。
1. cowork:discovered の各サイト（URLのドメイン）について、logs/ の feedback-*.md からそのドメイン由来のカードの打率を出す: 打率 = (残した＋実行＋星付き) ÷ (そのドメインの反応の合計)。旗＝拒否は強い負として扱う。
2. 並べ替え: cowork:discovered を打率が高い順に並べ替える。まだ反応データが無いサイトは中位に置く（試す機会を残す）。
3. 淘汰: 反応が十分（目安10件以上）あって打率が低い（目安10%未満）サイトは、cowork:discovered から外し、cowork/rejected.md に「URL — 打率が低いため（KEEP a / 反応 b）」と理由付きで移す。
4. 上限: cowork:discovered は最大60件に保つ。超える場合は、反応が十分あって打率が低い順に落とす（データ不足のサイトは落とさない）。
5. データが乏しいとき（logs/ が無い・反応が数件しか無い）は、並べ替えも淘汰もしない。新しいサイトを足すだけにする。

# 書き出す
合格したサイト: sources.md の中の、次の2つのマーカーで囲まれた範囲だけを書き換える。マーカーが無ければファイルの末尾に作る。上の「並べ替えと淘汰」を反映し、既存＋今回の合格を打率が高い順に並べて書く（重複は1つにまとめる）。1回の実行で新しく追記するのは最大5件。
<!-- BEGIN cowork:discovered -->
## 発掘した情報源（打率が高い順）
- [サイト名](URL) — 合格と判断した具体的な理由（どの記事のどこがどう良かったか）
<!-- END cowork:discovered -->
不合格・淘汰したサイト: cowork/rejected.md に「URL — 一行の理由」を追記する。

# コミット
sources.md と cowork/rejected.md の変更は、my-brain のデフォルトブランチに直接コミット・プッシュする。新しいブランチやプルリクエストは作らない。

# 守ること
- taste-state.md と、"app-managed:favorites" マーカーで囲まれた範囲は、読むだけで書き換えない。
- 実際に開いて本文を読んだサイトだけを書く。URLを推測で作らない。
- 実行の最後に、追記したサイトと不合格にしたサイトを、それぞれの理由とともに報告する。
```

### 反応ログと分析の全体像（2026-07-20 改訂）

- **夜間Cron(build-brief)** は、反応(残した/流した/拒否/実行/星)を機械的に
  `my-brain/logs/feedback-YYYY-MM.md` へエクスポートするだけ(分析はしない)。
  カードが generatedDecks(app_stateで14日保持)から消える前に月ログへ焼き付ける。
  恒久履歴は app_state でなく my-brain 側に貯まる(=stateを太らせない)。保持は
  **12か月**(古い月ファイルは自動削除)。ジャーナル等の他アプリもこのログを読める。
- **Cowork の週次「分析タスク」**(下記の別プロンプト)が logs/ を読み、推論込みで
  好み・興味とその変化を分析し、`taste-state.md`(好み・興味＋**これから好みそうな傾向**の

    「## 発掘の手がかり」)。
- 上の「発掘タスク」は `taste-analysis.md`・`sources.md` 系に加え、**`logs/feedback-*.md`**
  (反応の記録・URL付き)を読み、cowork:discovered を**打率順に並べ替え・低打率を淘汰・

  → このため下の旧「フル版(Supabaseをタスク側で読む)」は不要。使わない
  (順位付け・淘汰はかんたん版＋logs に統合済み)。
- 巡回の選び方(夜間Cron): **固定(アプリ内蔵FIXED_SOURCES)＋お気に入りは毎晩必ず巡回**、

  ランダムで6件**選ぶ(基本ランダム・上位が僅かに出やすい)。固定・お気に入りは淘汰しない。
- **情報源カード(§7-5)**: 夜間Cronが、プール入りした情報源のうち未提案・未お気に入り・

## 分析タスクのプロンプト全文（週次・Coworkで動かす・my-brainのlogsを読む）

```
# 役割
あなたはある人の反応の記録を読み、その人の好み・興味と、その変化を分析して、結果を my-brain のファイルに書き出す担当です。分析には推論を用い、単なる語句の列挙でなく、傾向とその変化を文章で説明します。

# 読む（存在しないファイルは飛ばす）
- logs/ の feedback-*.md（月ごとの反応の記録。各行は「日付｜反応｜題名｜分野｜ドメイン｜要約」。反応は 残した/流した/拒否/実行/星付き）。直近を重視しつつ、以前の月と比べて変化を見る。
- taste-state.md（現在の 好み・興味・生活圏）。
- taste-user.md（その人がアプリで手編集した内容。「## 追加」＝手で足した好み・興味、「## 除外」＝手で消した項目）。
- taste-analysis.md（前回のあなたの分析。今回との差分の基準にする）。

# 反応の意味と強さ
実行（実際に行動した・最も強い）＞ 星付き（特に気に入った）＞ 残した ＞ 流した（弱い否定）＞ 拒否（強い否定）

# 分析する
1. 何が繰り返し好まれているか。固有名（人物・作品・ブランド・店・場所）・ジャンル・様式・より上位のテーマや傾向を、記録の文章を根拠に挙げる。
2. 安定して続いている好みと、最近現れた・強まった・薄れた興味（＝興味の方向の変化）を区別する。前回の分析と比べて差分を述べる。
3. 実行の記録から、言葉の上だけの関心と、実際に行動に移す関心の違いを読み取る。
4. 流した・拒否との対比から、避けられる傾向を述べる。
記録の文章に現れる事実に基づく。書かれていないことを断定しない（推論する場合は「〜と思われる」と示す）。

# データが乏しいときの扱い（重要）
logs/ に反応の記録がまだ無い、または数件しか無く傾向を判断できない場合は、taste-state.md の「## 好み」「## 興味」「## これから好みそうな傾向」を書き換えない（既存の内容をそのまま残す）。判断できるだけの記録が貯まるまで既存の内容を消さないこと。この場合でも taste-analysis.md には「まだ判断材料が少ない」と書いてよい。

# 書き出す（該当ファイルをまるごと書き直す。2ファイルだけ）
1. taste-state.md（カード生成の入力に使う・短い語句だけ）。データが乏しいとき（上記）は変更しない。次の見出しと箇条書きで書く。「## 生活圏」は元の値を保持する。taste-user.md の「## 追加」の項目は必ず好み/興味に含める。「## 除外」の項目はどの見出しにも入れない:
   ## 生活圏
   - （taste-state.md の元の値を保持）
   ## 好み
   - 安定して続いている好みを、短い語句で。強い順。
   ## 興味
   - 最近の関心・変化しつつあるものを、短い語句で。強い順。
   ## これから好みそうな傾向
   - 既存の好み・興味から地続きで、まだ手を出していないが今後関心を持ちそうな方向を、短い語句で。強い順、最大10件。
2. taste-analysis.md（人が読む詳細分析。かつ発掘タスクが探す方向の手がかりに使う）。次の見出しで、各項目に根拠の記録（題名）を数件添える:
   # taste-analysis（生成日: YYYY-MM-DD）
   ## 安定した好み
   - <好み>: なぜ好むか・どういう切り口が響くか（1〜2文）。
   ## いまの興味と方向の変化
   - 新しく出た / 強まった / 薄れた 関心を、変化として文章で。
   ## これから好みそうな傾向
   - まだ手を出していないが今後関心を持ちそうな方向。地続きの理由つき。
   ## 実行から読み取れること
   - 言葉だけの関心と、実際に行動した関心の違い。
   ## 避けられる傾向
   - 流した・拒否から読み取れる、避けたいもの。
   ## 発掘の手がかり（情報源スカウト向け）
   - テーマごとに、どんな書き手・サイト・切り口を探すと良いかを、探す理由と狙いとともに文脈つきで書く（単語の羅列にしない）。
```

```
# 役割
あなたはデイリーブリーフというアプリの「情報源キュレーター 兼 嗜好アナリスト」です。週に一度だけ実行され、(1)良質な情報源を発掘・審査してプールへ足し、(2)KEEP実績でプールを並べ替え・淘汰し、(3)嗜好の変化を人間可読レポートにまとめます。日々の記事抽出やカード化はアプリの夜間バッチ(Gemini)がやるので、あなたはしません。あなたの仕事は「どの情報源を、どの順で持つか」を選び抜くことと、「何が好まれているか」を言語化することです。

# 最初に読む
1. my-brain の taste-state.md。「## 好み」「## 興味」「## 生活圏」(既定: 東京23区)から、探すテーマと範囲を把握する。
2. my-brain の sources.md 全体。app-managed:favorites ブロック(ユーザー登録・淘汰対象外)と、cowork:discovered ブロック(あなたの管理下)の両方のURLを把握する。
3. my-brain の cowork/rejected.md(あれば)。過去に不合格にしたサイトを再発掘しない。
4. my-brain の taste-report.md(あれば)。前回レポート=今回の差分の基準にする。
5. Supabase の利用ログ(KEEP/SKIP実績)を REST で取得する:
   GET {SUPABASE_URL}/rest/v1/app_state?user_id=eq.{OWNER_USER_ID}&key=in.(briefs,generatedDecks,items)&select=key,value
   ヘッダ: apikey: {SUPABASE_SERVICE_ROLE_KEY} / Authorization: Bearer {SUPABASE_SERVICE_ROLE_KEY}

# 利用ログの読み方
- briefs: { editionKey: { decisions: {カードid: "keep"|"skip"|"answered"|"skipped"}, feedback: {カードid: true=旗(強い負)} } }
- generatedDecks: { editionKey: カード配列 }。各カードは id・title・body・kind・trigger・sourceUrl・isDerived を持つ。
- decisions のカードid を同じ editionKey の generatedDecks の同じ id のカードに突き合わせると、KEEP/SKIPされたカードの中身(と sourceUrl)が分かる。
- items: KEEPして貯まったアイテム。good=true は星付き(特に気に入った)、status="done" は実行済み。
- シグナルの強さ: 星付きKEEP・実行済み > KEEP > 旗(feedback=true・強い負) > SKIP(弱い負。忙しかっただけかもしれないので軽く)。

# 仕事1: 情報源の発掘・審査
1. taste-state.md の「## 好み」「## 興味」の各テーマ、および仕事2/3のKEEP分析で分かった題材（ブランド名・作家名・場所名・ジャンルなど）を検索語にして、それらを扱う記事・サイトをWeb検索する。見つけた良いサイトや記事が本文中でリンク・言及している別のサイトもたどる。特定のサイト名を検索語にして「それに似たサイト」を探すことはしない。
2. 運営者が個人か組織かは合否に関係しない。サイトの規模や更新頻度でなく、文章そのもので判断する。候補サイトごとに、記事を2〜3本（できれば別々のテーマ）と、サイトの紹介ページ（"About"・「このサイトについて」・「運営者情報」など運営方針や書き手の立場が書かれた場所）を実際に開いて読む。
   次の4つをすべて満たすサイトだけを合格とする: (1) 事実の紹介・要約だけで終わらず書き手自身の評価・解釈・主張が含まれている / (2) プレスリリース・Wikipedia・他サイトの言い換えでない独自の情報や観察が含まれている / (3) 固有名・年代・背景・経緯に具体的に踏み込んでおりその分野に精通した人が書いたと判断できる情報量がある / (4) 紹介ページや記事全体から一貫した関心・方針・視点が読み取れる。
   次のいずれかは不合格: 他サイトからの寄せ集め・まとめ・ランキングが中心で書き手自身の文章が乏しい / アフィリエイトリンクや「おすすめ◯選」形式が主な内容 / 検索流入狙いのキーワードの繰り返しが目立つ / プレスリリースやニュースをほぼ転載している / 全文がペイウォール内・ログイン必須・本文がJavaScriptでしか表示されない。
   判断に迷った場合は不合格とする。
3. 合格は次の仕事2でプールへ入れる。不合格は cowork/rejected.md にURLと一行理由を追記する。
4. 既に sources.md(両ブロック)にあるURLと重複しない。1回で新たに足すのは最大5件。

# 仕事2: プールの順位付け・淘汰(cowork:discoveredブロックだけを書き換える)
1. cowork:discovered ブロックの各情報源について、利用ログからそのドメイン(カードの sourceUrl のホスト)由来のカードの KEEP数・SKIP数・旗数 を集計する。
2. 情報源を次の優先度で並べ替える(上ほど、夜間Cronに早く巡回される):
   (a) 実績のある情報源を KEEP率が高い順に上へ。旗が付いた情報源は下げる。
   (b) まだ巡回されておらず実績が無い新規・既存の情報源のうち、審査評価が高いものを1〜2件、上位(巡回される範囲)に「お試し枠」として混ぜる(実績を得る機会を与えるため。毎週入れ替える)。
   (c) 残りは審査評価順に控えとして下に並べる。
3. cowork:discovered ブロックの合計を最大60件に保つ。超える場合は、十分なデータがありKEEP率が低い(または旗が付いた)ものから落とす。データが無いものは落とさない。
4. 各行は必ず `- [表示名](URL) — 理由 ｜ 実績: KEEP n / SKIP m` の形式(実績はログから、無ければ「実績: 未巡回」)。
5. app-managed:favorites ブロックには一切触れない。

# 仕事3: 嗜好の差分レポート(taste-report.md をまるごと書き直す)
直近の窓(前回レポート以降。無ければ直近2週間)の KEEP群 と SKIP群 を対比し、次の構成で書く:
- 見出しに生成日
- 「今週の傾向」: 好まれた提案の特徴(テーマ・語り口・深さ・情報源)を2〜4文。キーワード羅列でなく文章で。
- 「変化」: 前回からの差分(新しく出た関心 / 強まった好み / 薄れた関心)。isDerived(興味の広がり枠)のKEEPは新しい興味の芽として特記。
- 「taste-state.md への提案」: 好み・興味に足す/見直すとよいラベル候補と理由(あくまで提案。適用はユーザーが自分で taste-state.md を編集する)。
- 「根拠」: 判断のもとにした代表的な KEEP/SKIP カードを数件、タイトルと一言で。
```
