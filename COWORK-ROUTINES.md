# Cowork / Claude Code 定期タスク仕様（情報源の発掘・審査・順位付け + 嗜好レポート）

このファイルは、デイリーブリーフの「判断が要る・低頻度・クリティカルパス外」の
仕事を、週1の定期タスク(Cowork または Claude Code の Routine)に任せるための
**仕様書 兼 プロンプト全文**です。ユーザーはこの中の「週次Routineプロンプト」を
自分の定期タスクに貼って設置します。SYSTEM-DESIGN §7・§8 の設計をそのまま実装
したもの。

---

## どこで動かすか

- **コワーク / コード どちらでも動く**(両方 Anthropic のクラウドで定期実行が可能)。
- 推奨: **Claude Code の定期タスク**。理由は my-brain が git リポジトリで、
  clone→編集→commit→push がネイティブで確実、Supabase も Bash の curl で読める
  (コネクタ設定不要)。コワークで動かす場合は GitHub コネクタの接続が要る。
- スケジュール: **週1・オフ時間(例: 日曜 03:00 JST)**。クリティカルパス外。
  走らない週があっても、夜間Cron(Gemini)は現状の sources.md をそのまま巡回し
  続けるので、ブリーフは止まらない(graceful degradation)。

## このタスクが持つ責務

日次のブリーフ生成(巡回→抽出→分類→編成)はアプリ側 Gemini(Vercel Cron)が
毎日やる。このタスクはそこに**乗せる情報源の質と並び**、および**嗜好の
振り返り**だけを持つ。

1. **情報源の発掘・審査**(§3-4/§3-7): 好み・興味に合う質の高いサイトを探し、
   審査し、合格分を情報源プールへ足す。
2. **情報源プールの順位付け・淘汰**(§7-4): KEEP実績で情報源を並べ替え、
   上限(〜40)を超えたら実績の低いものを落とす。お気に入り(ユーザー登録)は
   淘汰しない。
3. **嗜好の差分レポート**(§7-6/§8): KEEP/SKIPログを分析し、好み・語り口・
   深さの傾向と変化を人間可読レポートにして書き戻す(Obsidianで読む)。
   taste-state.md の好み・興味は書き換えない(確定はユーザー本人)。

## 必要なアクセス・環境変数(タスク側に設定)

- my-brain リポジトリの git 読み書き(本人のリポジトリ)。
- Web 検索・閲覧(発掘・審査用)。
- Supabase(**読み取りのみ**): `SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` /
  `OWNER_USER_ID`。

## 扱うファイル(my-brain)

- `taste-state.md`(**読むだけ**): 「## 好み」「## 興味」「## 生活圏」「## 願い」。
- `sources.md`: 情報源リスト。**アプリ管理の
  `<!-- BEGIN app-managed:favorites -->`〜`<!-- END app-managed:favorites -->`
  ブロックには絶対に触れない**(お気に入り=ユーザー登録・淘汰対象外)。
  このタスクが管理するのは下記 Cowork ブロックだけ:
  ```
  <!-- BEGIN cowork:discovered -->
  ## 発掘した情報源（KEEP実績順）
  - [サイト名](https://example.com) — 一行の採用理由 ｜ 実績: KEEP 3 / SKIP 1
  <!-- END cowork:discovered -->
  ```
  アプリの夜間Cronは sources.md 内の全箇条書きURLを読むので、このブロックの
  URLも自動で巡回対象になる(アプリ改修不要)。
- `cowork/rejected.md`: 審査で不合格にしたサイトのURLと理由(再審査回避)。
- `taste-report.md`: 嗜好の差分レポート(このタスクが毎回まるごと書き直す)。

## ★コワークにはGitHubコネクタが無い場合の半自動版（2026-07-20 実機で判明）

コワーク(コネクタ方式)には GitHub コネクタが用意されていないことがある
(実機で確認: Gmail/Calendar/Drive/Spotify/TickTick のみ、GitHub検索でも出ない)。
その場合コワークからは my-brain に直接書けない。**git書き込みを諦め、コワークは
「発掘して一覧で出す」だけにして、登録はアプリの「設定 → お気に入りの情報源」
から人手で行う**(この欄は元々 my-brain の sources.md へ同期され夜間生成に使われる)。
GitHub連携ゼロで今すぐ回せる。フル自動(git直書き)がしたい場合は、コネクタでなく
git ネイティブな **Claude Code(コード)側でタスクを作り、my-brain をリポジトリに
追加**する(下の「どこで動かすか」参照)。

# 利用ログの読み方
- briefs: { editionKey: { decisions: {カードid: "keep"|"skip"|"answered"|"skipped"}, feedback: {カードid: true=旗(強い負)} } }
- generatedDecks: { editionKey: カード配列 }。各カードは id・title・body・kind・trigger・sourceUrl・isDerived を持つ。

- items: KEEPして貯まったアイテム。good=true は星付き(特に気に入った)、status="done" は実行済み。
- シグナルの強さ: 星付きKEEP・実行済み > KEEP > 旗(feedback=true・強い負) > SKIP(弱い負。忙しかっただけかもしれないので軽く)。

## アプリ側の次ステップ（このタスクだけでは埋まらない穴）

上のタスクは sources.md を「KEEP実績順に並べ、〜40件に保つ」まではやるが、
**夜間Cronは今、プールの上位数件しか巡回しない**(`lib/briefPipeline.ts` の
`SOURCE_LIMIT` と本文予算 `TOTAL_TEXT_LIMIT`)。40件ストックしても下位は
巡回されず KEEP データが付かない=浮上できない。プール全体をまんべんなく
活かすには、夜間Cronの巡回対象を「毎回ローテーションで違う数件も覗く」
=§7-4 のバンディット探索をアプリ側に入れる小改修が必要(トークン予算と
両立させるため、単純な `SOURCE_LIMIT` 引き上げではなく巡回対象の
ローテーション選択で実装する)。**次のステップとして別途実施する。**

それまでの当面の挙動: 上のタスクが毎週「お試し枠」を上位に差し込むことで、
巡回される範囲に新しい情報源を計画的に露出させ、擬似的に探索を回す。
