## 1. 前セッションからの経緯（要約）

v19プロトタイプのNext.js App Router + TS移植から始まり、長いUI磨き込みセッションを経て
以下の大きな仕様変更がユーザーとの対話で確定した:

1. **Keep(場所)とMediaRecord(作品)の2コンテナを廃止し、単一の `Item` に統一。**
   「場所か作品か」は排他ではなく「種類(kind) × 場所の有無(area)」の直交軸。
   例: 新作映画=movie+area(劇場)、旧作映画=movieのみ(家で観る)、そこでしか買えないモノ=thing+area。
2. **分類の最上位は「願望の究極の対象物」による4ドメイン**:
   モノ(thing)・バショ(place)・タイケン(experience)・ジョウホウ(info)。
   ストック/プラン/アーカイブの棚はすべてこの4区分・この語彙で統一。
   カード上の種類ラベルもドメイン名を表示（ゴールだけは別枠）。
   展覧会・ライブは「その場でしか体験できない」のでタイケン所属（作品扱いをやめた）。
3. **ウィッシュはストックのカテゴリではなく上流の「受信箱」。**
   タブバー右の独立した丸ボタン(✨)からどのタブでも書ける。書く時に4ドメインを選択。
   ウィッシュはブリーフの生成材料になり、KEEPされて初めてItem(origin:"wish"、WISHバッジ付き)になる。
   アーカイブ最下部に全ウィッシュのフラットなチェックリストを置き、派生カードが
   バインドされたら自動でチェックが付く(`isWishBound`、手動トグル不可)。
   チェック済みのウィッシュ詳細では「叶えた！」「ゴールにする」を非表示。
4. **初期案の「マガジン(雑誌)風」というアイデアは廃止済み**(ユーザー明言、2026-07-12)。
   現行のデザイン言語は幾何学図形を使ったバウハウス調(§5参照)。初期要件定義書の
   「洋雑誌スタイル」の記述は古い。コード上の `magazine`(その日のプランを表す
   データ名)・`buildMagazine` は初期案の名残の内部名で、UI概念としての
   「マガジン」はもう存在しない(実体は「今日のバインダー」)。改名は
   影響範囲が広いため未実施。

## 3. データモデル（lib/types.ts が正）

- `ItemDomain = "place" | "experience" | "info" | "thing"`
- `ItemKind = place | exhibition | live | activity | food | movie | book | album | info | thing`
  （各kindは `lib/constants.ts` の `KIND_DOMAIN` で必ず1ドメインに属す。
  分類は `domainOf(item)` (lib/helpers.ts) が唯一の入口）
- `Item`: kind / title / creator? / category?(自由文の元カテゴリ) / **area?(場所の有無は
  ドメインと独立の直交軸)** / status: candidate→planned→done / addedAt / doneAt? /
  expiresAt? / price? / images? / meta? / sourceUrl? / color? / good? /
  **origin: "brief"|"manual"|"wish"**（バッジ表示 KEEP/WISH/なし） / sourceWishId?
- `Wish`: title / **category: ItemDomain** / status: stock|fulfilled / addedAt / fulfilledAt?
- `hasPlace(item)`: area有無の判定。地図のピンは**ドメインを問わず**これで出す。
- `isWishBound(wish, items)`: 派生Item(sourceWishId)が candidate 以外なら true。
- `BindLogEntry`: `id / boundAt / items(id・title・kind・color・imagesのスナップショット) /
  undone / undoneAt?`。`registerBinder`(確定ビューのバインド！)がdoneへ確定した
  Itemをまとめて1件記録する(`AppState.bindLog`)。プロフィール(設定)画面から
  一覧・元に戻す(doneをcandidateへ戻すだけの単純な取り消し、マガジンの再構築はしない)。
- その他: goals(checkIns、checkin 14日毎 / milestone 45日毎)、magazine(その日のプラン、
  日付が変わると pendingReview へ)、briefs(朝刊am/夕刊pmの2エディション/日)、
  `shelfOrder: Record<string, string[]>`(アーカイブの棚を長押しドラッグで
  並べ替えた結果。棚の識別子をキーに、その棚のBinderShelfItem.keyを並び順
  どおりに並べた配列。2026-07-12追加、詳細は§4のアーカイブの項参照)。

## 4. タブ構成（components/AppShell.tsx）

ナビ順: **アーカイブ(records・初期タブ) / ブリーフ(brief) / ゴール(goals) / ストック(stock) / プラン(execute)**。
ピルの右に独立した✨ボタン=ウィッシュ入力(`AddWishSheet`)。プロフィール(設定)は
ヘッダーの丸ボタン(`lucide-react`の`Settings`アイコン、2026-07-12にUserアイコンから変更)。

- **ブリーフ**: カードデッキをスワイプ(右=KEEP/左=SKIP)。育成カード(checkin/milestone)は
  スワイプ無効でフッターの「あとで/記録する」ボタン。KEEPでItem生成
  (sourceWishTitleが未達ウィッシュと一致すれば origin:"wish")。
- **ストック**: 候補(status==="candidate")のItemだけを4ドメインの `CardStack` で表示。
  追加シートはドメインごと(バショ=URL解析(現状モック)、タイケン/ジョウホウ=`AddKindItemSheet`、
  モノ=`AddThingSheet`)。カード詳細オーバーレイに「＋プランに追加 / 行った系 / 削除」。
- **プラン**: 地図(hasPlaceなItemのピン) + 「今週のおすすめ」エンベロープ(地図座標の近接
  クラスタリングによるモデルプラン。面に行き先の箇条書き、右上に開かず選べる+ボタン、
  タップで詳細シート) + 4ドメイン棚。表示対象はストックと同じく候補のみ
  (「選び直す」でバインダーを開き直している間だけ、綴じてあるカードも編集用に見える)。
  選択はAppShellの共有state(`PlanSelectionBar`、タブ横断)。
  **バインド！の意味(§7.8で確定した正)**: バインド！=「これらを実行した」という
  実行済みの確定。選択→バインド！(`buildMagazine`、planned化)→確定ビュー
  (ConfirmedStack)で内容を確認→下部のバインド！(`registerBinder`)で綴じると
  **全カードがdoneになり**バインダーごとアーカイブへ移る。バインドしたカードは
  ストック/プランの両方から消える。タスク管理的な「あとでやる予定リスト」では
  ないことに注意(§7.8の経緯参照)。バインドは取り消せる: `registerBinder`が
  `BindLogEntry`を記録し、プロフィール(設定)画面の「バインドの記録」から
  誤バインドを個別に元に戻せる(2026-07-12追加)。
  地図(`MapCanvas`)はAppShellの共有スクロールに乗ったまま`position:sticky; top:0`
  で画面上部に張り付き、下の棚をスクロールしても地図だけは常に見える
  (2026-07-12追加)。stickyで張り付いた瞬間(IntersectionObserverでセンチネル
  要素の可視/不可視を検知して判定、`mapStuck`)に地図の幅を100%→62%へ
  アニメーション(width遷移。transform:scaleだと縮んだ絵の周りに元サイズ分の
  空白が残ってしまうため、実際にレイアウトへ効くwidthを使っている)。
  地図右下に全画面トグル(`Maximize2`/`Minimize2`)があり、`MapFullscreenOverlay`
  としてcreatePortalでdocument.body直下へ描画する(sticky親が新しい重なり
  コンテキストを作るため、中でposition:fixedのままだとnav(zIndex:25)より
  手前に出せず閉じるボタンが押せなくなる。バインド！ボタンの時と同じ理由・
  同じ対処)。開閉ともscale+opacityでズームアニメーションする
  (`BottomSheet`と同じ「mount直後にrAFでenteredをtrue、閉じる時はまず
  falseに戻してから実際にアンマウント」のパターンに統一)。
- **アーカイブ**: 実行済み(done)がバインダー棚に積み上がる。棚=バショ(エリア毎)/タイケン・
  ジョウホウ(種類毎)/モノ(巻ごと、§5参照)/ゴール + 日付ビュー。最下部にウィッシュの
  チェックリスト(§1-3)。**リストビューの5棚(バショ/タイケン/ジョウホウ/モノ/ゴール)は
  それぞれ長押しドラッグで並べ替えられる**(`BinderCoverflowRow`、2026-07-12追加)。
  長押し(450ms、10px以上動くとキャンセルされ通常のスワイプ/スクロールに委ねる)で
  つまみ上げ、そのまま横へ動かすと隣と入れ替わる。並び順は`AppState.shelfOrder`
  (棚の識別子→keyの配列)に永続化。日付ビューは対象外(自動生成のため並べ替え不要)。
  日付ビューは月をまたぐと見づらいため、月ごとに行(BinderCoverflowRow)を
  分けて表示する(2026-07-12追加)。
- **ゴール**: バインダーグリッド + チェックイン記録。
- **プロフィール(設定)**: ヘッダーを「プロフィール/ABOUT YOU」から「設定/SETTINGS」へ
  改名(2026-07-12、アイコンがUser→Settingsに変わったことと整合させるため)。
  今、気になっていること / 興味・好み(自動検出、1色1サイズのチップに統一。
  以前はカテゴリ色+重み比例フォントサイズでバラバラに見えていた) /
  お気に入りの情報源 / バインドの記録(上記)。長い説明文を削り、
  情報源・バインドの記録の行(`SettingsRow`)・削除/元に戻すの丸いアイコン
  ボタン(`IconButton`、PlanSelectionBarの「選択を外す」と同じ語彙)を
  共通コンポーネント化し、セクションごとに実装がバラバラだった見た目を統一した。

## 5. デザインシステムの要点

- カード共通言語: `PosterCard`(パンチ穴 `PunchHoles`・バッジ/action/トグル/goodは
  右上に1本の横並びflexでまとめて詰める、2026-07-12変更。以前はバッジ左上・
  action中央・トグル右上の3点をspace-betweenで分散配置していたが、
  「バッジは基本右上に」「actionは＋トグルと同じ大きさのアイコンで
  コンパクトに」という指定で全部右詰めへ統一した。actionは文字ラベルでは
  なくチェックアイコンのみの20px円ボタン(＋トグルと同寸)。goodの星は
  従来top:8,right:8の独立配置だったが、バッジが右上に来たことで衝突する
  ため同じ列に合流させた)。穴と文字の回避マージンは `HOLE_CLEAR`(タイトル
  部分にのみ残る)。
- バインダー: `Binder.tsx` の Binder3D + `BinderCoverflowRow`。表紙はバウハウス的
  グリッド構図(45/90/180度・塗りつぶし図形のみ)。願望の4ドメイン+ゴールは
  それぞれ専用のデザインコード(Accentのkind)を持つ:
  target(ゴール、`goalAccent`)=全面べた塗り+上端中央からはみ出す円。
  geo(バショ、`placeAccent(エリア名)`)=上端の広い帯(58%)+無地下地+2x2グリッドで
  複数図形。背表紙(`BinderSpineFace`)は単色(2026-07-12修正、以前は45度の
  縞模様だったが「背表紙は単色に」という指定でgeo固有の縞処理を撤去し、
  target等と同じ単色+陰影オーバーレイのフォールバックに統一した)。
  media(ジョウホウのみ、`MEDIA_ACCENT[kind]`)=上端の細い帯(26%)+縦縞下地+
  隅寄せの図形ひとつ。side(タイケンのみ、`EXPERIENCE_ACCENT[kind]`、
  2026-07-12新設)=帯を上端ではなく**左端(蝶番側)に縦向き**で配置(幅34%)。
  図形は当初1個だけ帯の下端に寄せていたが、「正方形のグリッドで4〜5個
  縦に並べてほしい」という指定を受け、同じ図形(kind固定)をSIDE_MOTIF_
  COUNT(4)回、帯の全高に均等配置する総柄(トーテムポール状)に作り直した
  (2026-07-12続報、下地=帯の色自体は変更なし)。stamp(モノのみ、
  `thingVolumeAccent(巻数)`)=白地(PAPER)に色調違いの円を3つ組み合わせた
  構図(`StampMotif`、2026-07-12続報で3度目の作り直し。初版=円1つ全面
  べた塗り→2版=角の三角1つ全面べた塗り→現行=白地+複数円、という順で、
  「白地に、色のついた円を組み合わせて」という明確な指定に落ち着いた)。
  位置はタイトルと重ならないよう、CoverBodyのタイトルが絶対に伸びてこない
  上部の余白帯(spacer、上から34%)の中に収めている。個体ごとにハッシュで
  色を振る他の3つと違い、モノは「件数がTHING_ITEMS_PER_VOLUME(20件)を
  超えたら同じ意匠のまま色だけ変えて次の巻(Vol.2…)に分かれる」という
  運用(ユーザー指定)のため、色は「エリア名やkind」ではなく「何巻目か」
  という連番だけから引く。
  `BinderEdgeFace`(無地の側面、右端の厚み)には角丸を付けない(2026-07-12修正)。
  以前はCOVER_RADIUS(12px)を与えていたが、この面の幅は本の厚み(depth、細くて
  5px・GoalsTabのように太くても22px)しかなく、rotateY(90deg)でほぼ真横を
  向いた状態のままGoalsTab側のrotateY(-14deg)・perspectiveごしに描画される
  ため、角丸のカーブが表紙の丸い角のすぐ外側から槍のように尖った影となって
  突き出て見える描画崩れがあった(ユーザー指摘「右側の角丸に影が沿っていない」)。
  半径を幅の半分にクランプしても改善しなかった(深い3D回転と角丸の組み合わせ
  自体が原因)ため、この面は角丸を諦めて素の四角に戻した。表紙面は引き続き
  角丸を持つため、表紙の丸い角自体が欠けて見えることはない。
  →この対応は「影」の見た目だけの話で、**表紙自体のテクスチャが角丸の
  ところで透過していない(直角のまま突き出る)」という別の不具合が
  geo/media系のアクセント帯(triangle/circle等のAccentMotifグリッド)で
  残っていた(target系は角に不透明な塗りが来ないため気づかれなかった
  だけ)。原因はBinder3D側で表紙面(BinderCoverFace)を包む外側のdiv
  (box-shadowを持たせるためだけの層)に`overflow:hidden`+同じ角丸を
  重ねて付けていたこと。この外側divはtranslateZ/rotateYで3D変形されて
  おり(GPU合成レイヤーに昇格)、overflow:hidden+border-radiusのクリップを
  3D変形された要素自身に適用すると、Chromiumで角丸マスクの合成が
  正しく行われない。内側のBinderCoverFace自身(3D変形を受けない)が
  すでに同じ角丸+overflow:hiddenで自分の内容を正しくクリップしている
  ため、外側divのoverflowは`hidden`ではなく`visible`にした(box-shadowは
  要素自身のoverflowに左右されないため、影の見た目は変わらない)。
  実機での再現待ちだったこの手のバグは静的スクリーンショットの拡大比較
  (Python/Pillowでのcrop)で原因を特定できた。
- **zIndex規約**: navピル=25、nav手前の下地グラデーション=15、
  フローティング操作ボタン(バインド！2箇所・ブリーフのフッター)=**26**(グラデーションに
  覆われないため。この規約を破ると「ボタンが半透明マスクに隠れる」バグが再発する)。
- navはposition:sticky(fixedはiOS URLバー伸縮でズレるため禁止)。
- タブ切替は `key={tab}` でスクロールコンテナごと作り直し(スクロール位置引き継ぎバグ対策)。
  ブリーフタブ滞在中だけ外側コンテナのoverflowをhiddenにする。

## 7. ★未解決の最優先バグ: ブリーフタブのカード挙動（実機iPhone Safari）

**症状（ユーザー報告・実機スクリーンショットあり）:**

**2026-07-11セッション・続報(ユーザーの実機再テスト後):**
- svh化(上記)で**症状3(チェックインカード昇格時の縮み)は解消**と報告あり。
  一方で**症状2(スワイプ確定で次のカードが出る瞬間のガクつき)は残った**。
  この非対称な結果から、症状2はdvhのようなSafari固有の揺らぎではなく、
  **アニメーション設計そのものの欠陥**だったと判明した(ユーザー本人の指摘)。
- **根本原因**: `peekTransform`(控えのカードの位置)は、指でドラッグしている
  間はdxに応じて追従するが、指を離してcommit()した後は**320ms後にindexが
  実際に進む瞬間まで**8px下にずれた位置のまま停止していた。その320ms後、
  「控え→本番」に役割が切り替わるのと**同じReactコミットの中で**初めて
  transformの値を(0,0)へ動かしていたため、CSSトランジションが正しく
  発火する保証がなく瞬間移動して見えていた(役割切り替えという離散的な
  イベントに、連続的な位置合わせのアニメーションを乗せてしまっていたのが
  設計上の誤り)。ChromiumでもmouseイベントでSWIPE_THRESHOLDを正しく
  超えさせて詳細に計測すると、controlled雑な検証では見逃していたが、

  見落としていたことが判明)。

  `exit`ステート(=手前のカードが飛び始める、まさにその瞬間)に連動させた。

  定位置(translateY 0, scale 1)へ向けて動き始めるようにしたため、320ms後に
  実際にindexが進んで「控え→本番」に役割が切り替わる頃には、transformの値は
  既に定位置に収まっている。役割切り替えの瞬間には見た目上なにも変化しない
  ため、原理的にガクつきようがない(`components/tabs/BriefTab.tsx`の
  `peekTransform`/`peekTransition`定義とその上のコメント参照)。
- 検証: Playwrightでmousedown→複数回のmousemove→mouseupにより実際に
  SWIPE_THRESHOLDを超えるドラッグを再現し、release後40ms刻みでtop/peek両
  要素のtransform・transition・boundingBox.topを記録。控えカードのtopが
  211px→199px→…→191pxと連続的に収束し、320ms後の役割交代の瞬間には
  既に191pxで一致していることを確認(離散的な瞬間移動が消えたことを数値で確認)。
  `tsc`/`eslint`/`build`も通過。**この修正はChromium上の実測データに基づく
  確認が取れているため、前回のdvh仮説よりも確度が高い。** とはいえ実機
  Safariでの最終確認はまだ。

## 7.5 ストック/プランタブのバインド関連バグ（2026-07-11、修正済み）

ユーザー報告(実機スクショあり):
1. バインド！したカードがストック/プランタブの候補から消えない
2. バインド！ボタン・育成カードのフッターに半透明の靄がかかる
3. プランの確定ビュー(バインダー)で下にスクロールすると「← 選び直す」より
   上のヘッダーが画面外へ流れて隠れる
4. 「← 選び直す」で地図に戻り選択を編集して再度バインド！しても、確定
   ビューに戻らず地図のまま(変更が反映されて見えない)

**すべて原因を特定し修正済み:**

1. `StockTab.tsx`の候補一覧が`status !== "done"`でしか絞っておらず、
   バインド済み(`status: "planned"`)のItemも表示していた。`status ===
   "candidate"`のみに変更。
2〜3. **1つの根本原因に起因**: 実行タブの確定ビューの「バインド！」ボタン
   がAppShellの`key={tab}`配下に生えており、その先祖のどこかがzIndexを
   閉じ込める重なりコンテキストを作ってnav手前のグラデーションより手前に
   出せなくなっていた(2)。この対策として`ExecuteTab.tsx`のバインド！
   ボタンをReact Portalで`document.body`直下へ描画先ごと移し、祖先の重なり
   コンテキストの影響を完全に排除した。
   3の方はさらに厄介で、**修正が2回目の修正によって毎回上書きされて無効化
   されていた**: AppShell側で「実行タブの確定ビュー表示中は外側のタブ
   スクロールをロックする」ロジック(`scrollLocked`、ブリーフタブと同じ
   仕組みをここにも拡張)を正しく実装したにもかかわらず、実機で直っていない
   と報告された。調査の結果、`ExecuteTab.tsx`の`ConfirmedStack`が既に
   持っていた`killMomentumScroll`という別のヘルパー(スクロール位置を
   0へ戻すためのもの)が、外側のコンテナに対しても呼ばれており、この関数は
   末尾で無条件に`overflowY = "auto"`へ書き戻す実装だったため、AppShell側が
   設定した`hidden`を画面表示直後に毎回上書きして壊していた。
   `data-scroll-locked`属性を仕込んでReactの計算値(`true`)とDOMの実際の
   値(`auto`)がズレていることを確認し、原因を特定。`killMomentumScroll`を
   外側のタブスクロールコンテナに対して呼ぶのをやめ(内側の`scrollRef`だけ
   対象にする)、ロックはAppShell側に一元化した。
4. `mapMode`(地図/確定ビューのどちらを見せるか)が`ExecuteTab`のローカル
   stateだったため、既に実行タブにいる状態でバインド！を押しても
   `setTab("execute")`がno-opでコンポーネントが再マウントされず、
   `mapMode`がtrueのまま残っていた(マガジン自体は裏で正しく更新されて
   いた)。`mapMode`をAppShellへ`execMapMode`として引き上げ(2の
   scrollLockedの判定にも必要だったため一石二鳥)、マガジンの
   `decidedAt`が変わるたびに`setMapMode(false)`する`useEffect`を追加。

**教訓**: zIndexの数値を上げるだけの対症療法は実機Safariで繰り返し破綻した
(このセッションだけで2回)。今回はPortalによる祖先非依存の描画と、
「ロックは1箇所が一元的に持つ」という設計原則の徹底で決着させた。同種の
「直したはずなのに直っていない」不具合が今後出た場合、まず疑うべきは
「別のコードが同じDOM要素のstyleを裏で上書きしていないか」。

## 7.6 「選び直す」流れ・「バインドしたカードがストック残留」の根本原因（実機Safari・ホーム画面PWA、2026-07-11続報）

§7.5の修正後もユーザーの実機(Safari、ホーム画面に追加したスタンドアロンPWA)
では**再現した**という報告を受け、改めて調査。この環境ではWebKitが
インストール不可のため実機での直接検証はできず、コードの静的解析と
Chromiumでの間接検証のみで原因を特定した。

**バグA: バインドしたカードがストックに残り続ける。**
`ExecuteTab.tsx`に、**全く別の操作をする2つのボタンが同じ「バインド！」
という文言を共有していた**ことが直接原因。
- AppShell側の`bindSelection`(タブ横断のフローティングUI、PlanSelectionBar
  や確定ビューへの遷移直後に押す方)は選択をその日のプランへ確定する操作
  (`buildMagazine`。対象Itemを`status:"planned"`にする)。
- 確定ビュー(`ConfirmedStack`)側の下部固定ボタンは`handleRegister`→
  `registerBinder`を呼ぶ、**全く別の**「今日のバインダーを閉じる」操作。
  `registerBinder`は仕様として意図的に「まだ`done`になっていないItemは
  `candidate`へ戻す」処理を行う(バインダーに残った=行かなかった候補は
  ストックへ差し戻す、という正しい仕様)。
  ところがこのボタンの文言も従来「バインド！」だったため、ユーザーが
  選択確定後の自然な流れとして確定ビューのこのボタンを押すと、
  「バインドしたはずなのに(実際にはバインダーを閉じただけで)Itemが
  candidateに戻ってストックに現れる」という体感になっていた。
  ロジック自体は仕様通りで、バグではなく**UIの文言による事故**だった。
  `git log`のコミットコメント自体もこの2つを「バインド！」と同一視して
  記述しており、少なくとも今回の実装セッション中はこの区別を意識せずに
  作業していたことが伺える。
- **対応(一度実施→ユーザー指示により撤回)**: `registerBinder`側のボタン
  文言を「今日を登録する」に変更し一度デプロイしたが、実機で試したユーザーから
  「バインドしても消えない」症状は解消しておらず、かつ**文言変更を無断で
  行ったことについて明確な抗議**を受けた。「バインド！」の文言へ戻した
  (2026-07-11)。以後、デザイン・文言の変更は事前にユーザーへ確認してから
  行うこと。
  文言を分けても症状が直らなかった事実から、原因はラベルの取り違えだけ
  ではない可能性が高い。有力な仮説: 確定ビューの下部には常時目立つ黒い
  ボタン(バインド！/registerBinder)があり、選択確定の直後にユーザーが
  自然な流れでこれも押してしまうと、たとえ文言が分かれていても
  「未実行のItemをcandidateへ戻す」という(仕様通りの)副作用が発動し、
  結果的に同じ体感(ストックにカードが残る)になる。この画面構成自体
  (「選択確定→即座に同じ場所に『今日を締めくくる』操作が常設されている」)
  に事故を誘発する設計上の問題がある可能性があり、**次セッションはここを
  疑うこと**。ただし憶測での再変更はユーザーの指示に反するため、まず
  ユーザーに再現手順(バインド！を押した直後か、確定ビューの別のボタンも
  押したか)を確認してから対応する。

**バグB: 「選び直す」ボタンがスクロールで画面外へ流れる(§7.5点3の再発)。**
`app/globals.css`の`html, body`が**縦方向のoverflowを実は一度も
明示的にhiddenにしていなかった**ことが根本原因。従来あったのは
`overflow-x: hidden`と`overscroll-behavior-y: none`のみで、後者は
「スクロール境界でのバウンス連鎖」を止めるだけであり、「文書自体が
そもそもスクロールできる」ことは防げない。AppShell.tsx側のコメントには
「ウィンドウ自体は絶対にスクロールしない」という設計意図が明記されて
いたが、その保証は内側の`data-tab-scroll-root`div止まりで、**html/body
そのものには及んでいなかった**。実機Safari(特にスタンドアロンPWA表示)で
box-modelの端数計算によりbodyの実コンテンツ高が100svhをわずかに超えると、
内側の各divが正しくoverflow:hiddenで自衛していても無関係に**html/body
自体がスクロールしてしまい**、ヘッダーごとページ全体が引きずられて
見える。Chromiumではこの端数のズレが起きなかった(または起きても
無視できる範囲だった)ため、これまでのChromium検証では再現しなかった
と考えられる。
- **対応**: `html, body`に`overflow: hidden`を明示。これにより文書自体が
  物理的にスクロールできない状態を構造的に保証する(overscroll-behaviorは
  引き続き多重の防御として残す)。
- **あわせて実施したUI再設計**: `ExecuteTab.tsx`の「← バインダーに戻る」
  「← 選び直す」を、地の文に紛れる素のテキストリンクから、nav・statチップと
  同じ「PAPER地+SOFT_SHADOWで浮く丸チップ」語彙のボタンへ作り直した
  (`backChipStyle`)。
- **一度追加→実機確認の結果撤回したもの**: Masthead込みでこのヘッダーを
  `position:sticky; top:0`の専用枠にまとめる案を「外側のロックが将来また
  壊れた場合の保険」として追加had。ところが実機ユーザーからのスクリーン
  ショットで検証したところ、これが**新しい不具合**を生んでいたことが
  判明: sticky化した瞬間、その下に不透明な背景を持つヘッダーが常に最前面に
  居座るため、ConfirmedStackのカードがスクロールでヘッダーの下を通過する際、
  カード上端(特に写真部分)がヘッダーの背景で不透明に切り取られて見える
  (sticky headerの仕様通りの挙動だが、カード側はその下に潜り込まれる
  前提のデザインになっていなかったため「見切れバグ」に見えた)。
  そもそもこのヘッダーはConfirmedStack自身の独立スクロール領域
  (`scrollRef`、overflow-y:autoで自分の内容を自分の箱にクリップする)の
  **外側**にあるので、スクロールする主体がConfirmedStackの内側だけで
  完結している限り、このヘッダーとカードが視覚的に重なることは構造上
  あり得ない。stickyは不要な追加リスクだったと判断し撤回、元の
  「普通のflowに置くだけ」の非sticky構成へ戻した(チップの見た目
  =backChipStyleはそのまま維持)。
- **教訓**: 「念のため」の保険的な変更(今回のsticky化)がユーザーに
  相談なく追加され、かつそれ自体が新しい実害のあるバグを生んだ。今後は
  「壊れていないものを保険目的で変えない」「変える場合は先にユーザーに
  意図を説明し許可を得る」を徹底する。
- **未検証事項**: WebKit実機での直接確認ができておらず(環境の制約は§7と
  同じ)、コード上の原因特定とChromiumでの間接検証(スクロール状態での
  ヘッダー・カードのbounding box比較)まで。次回は実機での再テストが必須。

## 7.7 「選び直す」画面の要件誤認と、バインド！が無反応に見える実タイミングバグ（2026-07-12続報）

§7.6のsticky撤回後もユーザーからNGが出て、実機スクリーンショット5枚を提供
してもらい判明した2点。

**要件の誤認: 「選び直す」はヘッダー固定ではなく、他タブ同様に流れて良かった。**
これまでの複数セッションを通じて「スクロールしてもMasthead・選び直すボタンを
画面上部に固定表示させ続ける」ことを正としてscrollLocked(§7.5)を組んで
きたが、これはユーザーの意図と違っていた。ユーザーの実際の要望は
「アーカイブ/ストック等の他タブと同じく、下までスクロールすればMasthead
ごと上へ流れて、カードが画面の一番上端まで届いてほしい」というもの。
実行タブの確定ビュー(ConfirmedStack)だけがMastheadを固定表示する特別な
構成になっていたため、他タブと挙動が揃わず「選び直すの下に境目ができて
カードが見切れる」という体感になっていた。
- **対応**: `AppShell.tsx`の`scrollLocked`から実行タブの特別扱いを削除
  (`tab === "brief"`のみに戻す)。`ExecuteTab.tsx`の`ConfirmedStack`が
  持っていた専用の入れ子スクロール領域(`scrollRef`、overflow-y:auto)を
  撤去し、他タブと同じくAppShellの共有スクロール(`data-tab-scroll-root`)
  に乗る普通のブロックにした。`handleRegister`(登録アニメーション前に
  リストを先頭へ戻す処理)は対象を`scrollRef`から
  `document.querySelector('[data-tab-scroll-root]')`へ変更。
  Chromiumで`data-tab-scroll-root`を最下部までスクロールし、「選び直す」
  チップのboundingBoxが画面外(y座標が大きく負)へ出ることを確認、また
  登録アニメーション(スタック→閉じる→落ちる→アーカイブタブへ遷移)も
  引き続き正常に動作することを確認した。
- **教訓**: 「ヘッダーを固定して見失わないようにする」という設計判断
  自体を、ユーザーに確認せず複数セッションにわたって前提にしてしまって
  いた。§7.5・§7.6の対症療法(sticky化など)はすべてこの誤った前提の上に
  積み上げられており、根から間違っていた。今回のように「他のタブと
  比べてこのタブだけ挙動が違う」という具体的な指摘は、抽象的な設計判断
  よりも信頼できる比較対象になる。

**「バインドしたカードがストックに残り続ける」の実タイミングバグ。**
ユーザー提供のスクリーンショットから、実際にバインドした4件のItemが
確定ビューには表示されつつ、その後ストックタブにも同じタイトルのまま
残っていることを確認。デモデータの重複投入(「デモ用データを投入」を
複数回押した?)を疑ったが、ストックの件数(14件)がデモデータセット中の
未実行(done:false)Itemの数(14件)と厳密に一致しており、重複はないと判明
(この時点で候補が本来10件に減っているはずが14件のままだった=バインドが
実質不発だったことを意味する)。
`buildMagazine`のロジック自体は個別カード選択・ストック棚からの選択・
今週のおすすめエンベロープ選択のすべての経路でChromium上で繰り返し検証し、
一貫して正しくstatusを`planned`に変える(ストックの候補数が正しく減る)
ことを確認済み。一方この検証作業中、**`BottomSheet`(エンベロープの詳細・
ストック棚の一覧などで使う共通オーバーレイ)を閉じた直後、画面下部の
バインド！ボタンを即座にタップすると、クリックが吸収されて何も起きない**
という再現性のある不具合を複数回引き当てた。原因は`BottomSheet.tsx`の
閉じるトランジション: `requestClose()`は`setOpen(false)`で背景のフェード
アウトを開始したあと、実際にこの要素がアンマウントされるまで220ms待つ
(トランジションを最後まで見せるため)。その220msの間もこの背景・カード
要素は`pointerEvents:"auto"`のままだったため、フェードアウト中の一瞬
だけ画面のどこをタップしてもこの見えなくなりつつある要素に奪われ、
奥にあるバインド！ボタンへ届かない。ユーザーが「エンベロープを選んで
シートを閉じて、すぐバインド！を押す」という自然な操作をした場合、この
220msの窓に入る可能性は十分にある。
- **対応**: `BottomSheet.tsx`の背景div・カードdivの両方で、
  `pointerEvents: open ? "auto" : "none"`に変更(閉じ始めた瞬間に
  タップを素通りさせる)。Chromiumで「シートを閉じるボタンを押した
  直後、待ち時間ゼロでバインド！をタップ」という最も厳しいタイミングを
  再現し、修正前は`force:true`でなければクリックが通らなかったのが、
  修正後は素のクリックで即座に成功することを確認した。
- **未確定事項**: これがユーザーの実機での症状と完全に一致するとは
  断定できていない(実機でのタップ順序を直接観測できないため)。ただし
  コード上明確に存在する再現性のあるバグであり、症状(「バインドした
  はずなのに何も起きていないように見える」)とも整合する。次回、実機で
  同様の操作(シートを閉じてすぐバインド！)をゆっくり・素早く両方試して
  もらい、再発するか確認すること。
  →(2026-07-12続報)このBottomSheetのバグは実在したが、「ストックから
  消えない」の主因ではなかった。真の根本原因は§7.8。

## 7.8 ★「バインドしたカードが消えない」の真の根本原因: バインド！の仕様理解が逆だった（2026-07-12解決）

ユーザーから実機スクリーンショット(magazineにIDが入っているのにカードが
candidateとしてストックに表示されている状態)の提供と、バインド！の意味の
明確な説明を受け、ようやく全体像が判明した。

**ユーザーが定義するバインド！の意味(これが正)**:
> バインド！はキープした様々な予定を実行した時に、バインダーにそれら
> 実行済みのカードをまとめてアーカイブするイメージで、確定したことを
> 確認するボタン。バインド！した=すでに実行されたタスクなので、カードは
> アーカイブにバインダーごとにまとめてアーカイブされ、ストックタブと
> プランタブからは削除される。

つまりこのアプリのプランタブは「あとでやる予定リスト」ではなく、
**「実行したことを綴じて確定する」流れ**である。

**一方、従来のコードは正反対だった**: `registerBinder`(確定ビュー下部の
バインド！)は「まだdoneになっていないItemを**candidateへ戻す**」実装
(=バインダーに残った=行かなかった候補はストックへ差し戻す、という誤った
仕様解釈)。そのため、選択→バインド！→確定ビュー→バインド！という自然な
2段階の操作を完了するたびに、綴じたはずの全カードがストックへ静かに復活
していた。**これが§7.5〜§7.7まで延々と再発し続けた「バインドしたカードが
ストックから消えない」の真の根本原因。** 過去セッションのChromium検証は
1段階目のバインド！までしか押さずにストックを確認していたため、
一度もこの復活経路を踏んでおらず、すり抜け続けていた。

さらに、プランタブの地図・棚の表示フィルタも`status !== "done"`で、
バインド済み(planned)のカードが地図に出続けていた。前セッションで
これを「編集用の作業台だから仕様」と説明したのは**誤り**(ユーザーから
明確に否定された)。

**修正内容(3点、コミット参照)**:
1. `registerBinder`: 綴じた全カードを`done`(doneAt=綴じた時刻)にして
   アーカイブへ。candidateへの差し戻しを廃止。
2. プラン地図・棚の表示フィルタを`status === "candidate"`に変更。
   唯一の例外は「選び直す」でバインダーを開き直している間で、現在の
   magazineに綴じられているカードに限り(外す/入れ替えの編集のため)表示。
3. AppShellの日付ロールオーバー(綴じられないまま日付をまたいだ
   magazineの解散)時に、planned→candidateへ明示的に戻す。地図がもう
   planned単体を表示しないため、戻さないとどこからも見えないゾンビに
   なるため。

**検証(Chromium)**: デモ投入→エンベロープ2つ選択→バインド！(planned=4、
ストック候補14→10)→選び直す(地図プール=候補10+バインダー内4)→再バインド→
確定ビューのバインド！(綴じ)→done 38→42・planned 0・magazine解散・
アーカイブタブへ遷移・ストック/プラン両方から消えたことを、localStorageの
状態遷移で数値確認済み。WebKit実機での最終確認は未(次回ユーザーに依頼)。

**教訓**: 複数セッションにわたり「どの画面で消えるべきか」という表示側
ばかりを直し続けたが、そもそも**ボタンの意味(データフローの終着点)の
理解が逆**だった。ユーザーから「あなたはこのアプリの思想を理解していない
かもしれない」と指摘され、アプリの使い方の説明を受けてようやく発覚した。
挙動の不具合報告が繰り返し「直っていない」となる場合、実装の細部ではなく
**自分が正としている仕様そのものをユーザーに確認する**こと。

## 7.9 ★ゴールバインダーの影が角丸に沿わない問題の根本解決(2026-07-12・box-shadow→filter:drop-shadow)

§5で解決したはずの「バインダーの表紙の角丸」関連の描画崩れが、3回目の指摘で
再度実機スクショ(`IMG_1401.jpeg`)とともに報告された。§5で直した2件
(`BinderEdgeFace`の槍状の影、アクセント帯テクスチャの角丸未透過)とは
**別の3つ目の症状**: 表紙の角自体は正しく丸いのに、**box-shadowが角丸を
無視した直線のまま**表紙の外にはみ出して見える(「角が丸いのに影が直線で
バインダーと合っていない」)。

**根本原因**: `Binder3D`の表紙シャドウ用ラッパーdiv
(`translateZ`+`rotateY`で3D変形され、GPUの合成レイヤーに昇格している)に
`boxShadow: SOFT_SHADOW`を与えていたこと。box-shadowは要素の
border-radiusから「幾何学的に」矩形+丸角の影を計算する実装のため、
3D変形されたレイヤーに対してChromium/WebKitがこれを正しく合成できず、
回転や角丸を無視した素の直線的な影がそのまま(2D平面の位置に)描画されて
しまう。§5で直した2件(BinderEdgeFaceの槍状の影・アクセント帯の角丸未透過)
と原因の系統が同じ:「box-shadow/overflow:hidden+border-radiusを3D変形
されたレイヤーに適用すると壊れる」という一貫したバグの3つ目の現れだった。

**対応(ユーザーの「根本からデザインをやり直して」という要望に応え、
パッチではなく仕組み自体を変更)**: `boxShadow`をやめ、
`filter: drop-shadow(...)`に置き換えた(`components/Binder.tsx`の
`Binder3D`表紙シャドウ用div)。drop-shadowは要素が実際に描画した
ピクセル(アルファ値のシルエット)を元に影を生成するため、角丸で
クリップされた実際の輪郭に正しく沿い、かつ3D変形後のレイヤーそのものに
対して合成されるため回転にも追従する。SOFT_SHADOWの値
(`"0 4px 16px rgba(28,28,30,0.07)"`)はbox-shadowとdrop-shadowで引数の
並びが同じなのでそのまま`drop-shadow(${SOFT_SHADOW})`として流用できた。
Chromiumで角丸のピクセル単位クロップ比較を行い、影が矩形のまま突き出す
症状が消え、角丸の輪郭に沿った柔らかい影になったことを確認した
(BinderCoverflowRow・GoalsTab両方の呼び出し元で共通に効く)。

## 7.13 ★ゴールバインダーの「直角の角のはみ出し」を解決(2026-07-12・BinderEdgeFaceの配色)

§7.11でBinder3Dへ差し戻した後も指摘が続いたため、AskUserQuestionで
症状・参考画像の有無・影響範囲を確認した上で、ユーザー提供の実機
スクリーンショット(`IMG_1401.jpeg`)をPython/Pillowでピクセル単位に
拡大して原因を特定した。

**症状**: 表紙の角丸のすぐ外側に、直角の角を持つ細い縦長の板が、上端が
尖った台形状に突き出て見える。§7.9で対応した「影が直線」とは別の症状。

**原因**: この縦長の板の正体は`BinderEdgeFace`(「無地の側面」、本の厚み
=リングの反対側の開く側面)だった。この面はrotateY(90deg)でほぼ真横を
向いた状態で描画されるため、平らな四角形でも遠近法(perspective)越しに
見ると台形状に投影される。これ自体は正しい3D投影で、崩れではない。

問題は**配色**にあった: 従来の実装は`background:
"linear-gradient(90deg, rgba(0,0,0,0.18), rgba(0,0,0,0.04))"`という、
黒を薄く重ねるだけの半透明な指定で、面自体は無色だった。この面の
「裏」には3D空間上何も無く、ページの生成り色の背景がそのまま透けて
見えるため、意図していた「奥まった暗い陰」ではなく、単に背景を薄く
灰色がからせただけの**明るいグレーの板**として描画されていた。
表紙(濃いアクセントカラー)のすぐ隣に、無関係な色の四角い板が独立して
見えることで、「角丸の表紙(テクスチャ)を素の四角い箱に貼り付けただけ」
という印象になっていた——これがユーザーが一貫して主張していた原因の
実体だった。対して背表紙面(`BinderSpineFace`、リング側)は元々
アクセントカラーで塗られており、無地の側面だけが「色を持たない」例外に
なっていたことも、この面だけが浮いて見える一因だった。

**対応**: `BinderEdgeFace`にアクセントカラー(`spineAccent.color`を
`shade()`で少し暗くした色)を土台として持たせ、その上に従来の黒
グラデーションを陰影として重ねる不透明な面に変更した
(`components/Binder.tsx`)。表紙・背表紙・側面が同じ色ファミリーで
繋がることで、側面は独立した灰色の板ではなく「この色のバインダーの
厚み」として読めるようになる。**形状(素の四角、角丸なし)は一切
変更していない**ため、§5で対応した「角丸+3D回転の組み合わせで槍状に
崩れる」不具合を再発させる要素はない(色の変更のみ、ジオメトリ・
overflow・border-radiusには触れていない)。

検証: `npx tsc --noEmit` / `npx eslint .`(既知の2件のみ) / `npm run build`
すべて通過。Chromiumでゴールタブのグリッドを再スクリーンショットし、
ユーザー提供のスクショと同じ構図をピクセル単位で拡大比較。側面が
表紙と同系色の濃い板として表紙の陰のようになじみ、無関係な灰色の
板には見えなくなったことを確認した。

## 7.14 ★ゴールバインダー: 側面の角を透過フェードでマスク(2026-07-12続報)

§7.13の配色対応後もユーザーからは「直っていない、意図していない箇所を
直している」と指摘があった。ユーザー自身の理解(3点)を提示してもらい、
1つずつ実装と照らして回答した上で、対応方針をすり合わせた:

1. 表紙(`BinderCoverFace`)は実際にoverflow:hiddenで丸くクリップされて
   いる実体であり、「四角いベースに丸い絵を貼っただけ」ではない → これは
   不正確、と回答。
2. 見えていた四角い板は表紙の「影レイヤー」ではなく、独立したDOM要素
   `BinderEdgeFace`(本の厚みの側面)である → これも不正確(ただし
   見た目が影のように暗かったため誤解も無理はない)、と回答。
3. 「角丸に合わせてこの側面をマスクし、はみ出す部分は背景を透過させる」
   なら直る → **方向性としては正しい**と回答。ただし過去(§5)に同じ
   要素へ`border-radius`を与えたところ、深い3D回転(rotateY 90度)との
   組み合わせで「槍状の影」に描画が崩れるChromiumの不具合が起きた経緯が
   あるため、border-radiusをそのまま使うと再発する懸念がある、と説明。

ユーザーから「それで一度やって」と、③の方向(border-radius以外の手段で
マスク)での実装許可を得たため着手。

**実装**: `border-radius`・`clip-path`・`mask-image`など「クリップ系」の
プロパティは3D変形されたレイヤーに適用すると壊れる実績が複数あるため
一切使わず、単純な**アルファ合成のグラデーション**だけで同じ見た目を
作った。`BinderEdgeFace`の最前面に、ページ背景色(`BG`)を上下端で不透明、
`COVER_RADIUS+4`px内側で透明にフェードする`linear-gradient`を重ねる
(`components/Binder.tsx`の`EDGE_FADE`定数・`BinderEdgeFace`)。これにより
表紙の角丸が始まる領域だけこの側面が背景色に「消え」、border-radius系の
プロパティを一切使わずに角丸に合わせてマスクしたのと同じ結果を、過去の
バグを再発させるリスク無しで実現した。

検証: `npx tsc --noEmit` / `npx eslint .`(既知の2件のみ) / `npm run build`
すべて通過。Chromiumで拡大スクリーンショットを撮り、側面の上下端が
表紙の角丸の輪郭に合わせて滑らかに透けて消え、直角の角の飛び出しが
無くなったことをピクセル単位で確認した。**WebKit実機での最終確認は
引き続き未実施**(環境の制約は§7と同じ)。

**教訓**: この一連のやり取り(§7.9・§7.11・§7.13・§7.14)を通じて、
ユーザー自身に技術的な仮説(「四角いベースに丸い絵」)を言語化して
もらい、それを実装と照合しながら1つずつ正誤を答える、というやり取りが
最終的に一番早く正しい原因(BinderEdgeFaceという別要素の存在と、その
形状が角丸に追従していないこと)にたどり着いた。曖昧な「直りました」
report を繰り返すより、ユーザーの仮説を尊重しつつ実装の事実で答える
方が、この手の「同じ場所で何度も直っていないと言われる」状況を打開する
近道になる。

## 7.15 ★ゴールバインダー: 3D箱を廃止、台形に傾いた平面カードへ(2026-07-12続報・解決)

§7.14の透過フェード対応にも「全然違います」と明確な却下があり、
ユーザーから方針そのものの転換が示された:「3Dにする必要はない。角丸が
ちゃんとマスクされているなら、平面のテクスチャ(表紙)を、現状と同じ
角度でパースの効いた台形に変形するだけでいい」。

**これまでの一連の対応(§5・§7.9・§7.13・§7.14)がすべて後手に回っていた
理由がここで判明した**: Binder3Dは表紙・背表紙・無地の側面・裏表紙の
4面を組んだ「箱」だが、ゴールタブは棚のコンベア(スワイプで表紙⇄背表紙を
連続的に回転で行き来する)を一度も使っておらず、静止したグリッドで
常に表紙面だけを正面寄りに見せているだけだった。にもかかわらず箱を
まるごと組み立て続けていたため、表紙以外の3面(特に無地の側面
=BinderEdgeFace)が「見えなくていいはずなのに、角度の関係で見えてしまい、
表紙の角丸と衝突する」という同じ種類の問題を形を変えて繰り返し発生
させていた。

**対応**: `GoalBinderCard`(`components/Binder.tsx`)を新設し、ゴール
タブの表示をBinder3D(箱)からこちらへ差し替えた。表紙面(`BinderCoverFace`、
自分自身のoverflow:hidden+border-radiusで正しく丸くクリップされる、
3D変形を受けない平面)だけを、drop-shadow付きの非クリップなラッパー
(`perspective`+`rotateY(-14deg)`、以前と同じ見た目の角度)で台形に
傾ける。背表紙・無地の側面・裏表紙は**最初から作らない**。角丸と衝突する
別要素が存在しないため、この種の描画バグが構造的に起こり得ない。

検証: `npx tsc --noEmit` / `npx eslint .`(既知の2件のみ) / `npm run build`
すべて通過。Chromiumで角の四隅すべてをピクセル単位で拡大確認し、直角の
はみ出し・別要素の衝突が無く、角丸に沿った柔らかい影が付いていることを
確認した。**WebKit実機での最終確認は引き続き未実施。**

**教訓**: §5から§7.14まで、「箱の各面をどう直すか」という実装の枝葉に
何度も入り込んでいたが、そもそも「ゴールタブに箱(3D構造)自体が
必要かどうか」という前提を疑うべきだった。ユーザーが最初から示唆して
いた「3Dにする必要はない」という言葉を、これまでは「デザインの好み」
の話として扱っていたが、実際には「この構造自体が繰り返しバグを生んで
いる根本原因を取り除く提案」だった。同じ場所で複数回パッチが失敗する
場合、次のパッチを考える前に「そもそもこの構造自体が必要か」を疑うこと。

## 7.16 ゴールバインダー: 影が全く出ていなかったのを修正(2026-07-12続報)

§7.15で「柔らかい影が付いていることを確認した」と書いたが、これは
全体を見た目で確認しただけで、ピクセル単位のサンプリングまでは
していなかった。ユーザーから「その背景に軽く他と同じように影をつけて
ください」と指摘を受けて改めてPythonで画像のピクセル値を直接読むと、
カード下端から背景色への遷移が**1ピクセルの境界で完全に不連続**
(影のグラデーションが一切無い)であることが判明し、`filter:
drop-shadow(...)`が実質何も描画していなかったことが分かった。

**原因**: `GoalBinderCard`の実装は、`rotateY`を持つ要素(親)と、
`filter: drop-shadow(...)`を持つ要素(その子、`position:absolute;
inset:0`)を分けていた。Binder3D側で影が正しく効いている実装
(§7.9)は、`filter`と`transform`が**同じ要素**に乗っている。この2つを
別要素に分離すると、`perspective`の入れ子の中では`filter`が描画されない
(ピクセルサンプリングで実測)。

**対応**: `filter: drop-shadow(${SOFT_SHADOW})`を、`rotateY`と同じ要素
(`GoalBinderCard`内の唯一の変形ラッパー)へ移し、余分な中間divを削除した。
これによりBinder3Dの前面と全く同じ「filterとtransformを同じ要素に乗せる」
構成になった。

検証: `npx tsc --noEmit` / `npx eslint .`(既知の2件のみ) / `npm run build`
すべて通過。修正前後でカード下端の同じx座標の縦方向ピクセル値を
Pythonで直接比較し、修正前は1px境界での不連続な色変化(影なし)、
修正後はカード色→中間色2段階→背景色という滑らかなグラデーション
(影あり)になっていることを数値で確認した。

**教訓**: 「見た目のスクリーンショットで確認した」は「効果が実際に
出ている」ことの証明にならない場合がある(特に微弱な効果や、意図した
効果がゼロになっている場合は目視では気づきにくい)。影・グラデーション
など微妙な視覚効果を検証する際は、可能な限りピクセル値を直接サンプリング
して数値で確認すること。

## 7.18 ★ゴールバインダーの影、実機Safariでは依然出ず→box-shadowへ変更(2026-07-12続報)

§7.16でfilterとtransformを同じ要素に乗せる形にし、Chromiumでピクセル
サンプリングにより影のグラデーションを確認したが、ユーザーから実機
Safariでは影が出ていないという報告があった。

**判断**: `filter: drop-shadow()` + `perspective`(祖先)+ 3D transform
という組み合わせは、このコードベースで何度もブラウザ間の差異
(Chromiumでは効くがSafariでは効かない/崩れる)を起こしてきた前例が
多い(§5のBinderEdgeFace槍状の影、アクセント帯の角丸未透過など)。
`GoalBinderCard`の一番外側の箱(`perspective`を持つdiv)自体は、
`perspective`プロパティを持っているだけで、それ自身が3D変形されて
いるわけではない(3D変形されるのはその子)。つまりこの外側の箱は
普通の(3D変形を受けない)2D要素であり、ここに素の`box-shadow`を
使えば描画エンジンの違いに左右される余地がなくなる。

**対応**: `filter: drop-shadow(...)`をやめ、外側の箱(perspective を
持つdiv)に`boxShadow: SOFT_SHADOW`と、表紙の開く側に合わせた角丸
(`borderTopRightRadius`/`borderBottomRightRadius: COVER_RADIUS`)を
直接持たせた(`components/Binder.tsx`)。影の形は外側の箱自身の角丸に
沿うため、厳密に台形の輪郭に一致するわけではないが、傾きが浅い
(-14度)ため見た目には十分自然。表紙の実際のクリップは引き続き
`BinderCoverFace`自身(3D変形を受けない)が正しく行っている。

検証: `npx tsc --noEmit` / `npx eslint .`(既知の2件のみ) / `npm run build`
すべて通過。Chromiumでカード下端のピクセル値を再サンプリングし、
影のグラデーションが出ていることを確認。box-shadowは3D変形を受けない
要素への適用のため、Safari側でも(filterのケースと異なり)描画エンジンの
差異による失敗リスクは構造的に低いと考えられるが、**実機Safariでの
最終確認は引き続き必要**。

## 7.19 ★バインドのスタックzIndexを再修正: 表紙とカードの重なり順を分離(2026-07-12続報)

§7.17でzIndexを`i === 0 ? items.length : i`(先頭カードだけ常に最前面、
他はiが大きいほど手前)に変更したが、ユーザーから「直っていない。下から
スライドしてきたカードがどんどん一番上のカードの上に重なっていくような
アニメーションにしてほしい」と再度指摘があった。§7.17の修正は「先頭
カード(i===0)は常に例外的に最前面」という制約を残しており、これが
「一番上のカードの上にも重なっていってほしい」という要望と矛盾していた
(先頭カードだけは他のどのカードよりも常に手前のままだったため)。

**根本原因**: 先頭カード(i===0)を特別扱いしていたのは、`BinderFrontCover`
(閉じる表紙)が同じラッパーdiv内に同居しており、「表紙は閉じる時に必ず
束の一番手前に来る」という別の制約と、「カードの重なり順」という制約が
同じzIndexの値を取り合っていたため。カードの重なり順をユーザーの要望
通り(先頭も含めて素直にiが大きいほど手前)に変えると、表紙を閉じる時に
最後尾のカードより手前に出せなくなってしまう。

**対応**: `BinderBackPanel`(裏表紙)と`BinderFrontCover`(表表紙)を、
map内の各カードのラッパーから完全に切り離し、mapの外に独立した絶対配置
要素として置いた(`components/tabs/ExecuteTab.tsx`)。位置はitem[0]
(常に動かない=stackOffsetが常に0の位置)のボックスに一致させる
(`CARD_HEIGHT`をモジュールレベルの定数として新設)。zIndexは:
裏表紙=0(常に最背面)、カード群=`i + 1`(先頭カードも含めて素直にiが
大きいほど手前、特別扱い無し)、表表紙=`items.length + 2`(常に最前面)。
この3層を完全に独立させたことで、「カードの重なり順」と「表紙が閉じる
時は必ず一番手前」という2つの制約が別々のzIndex空間を持つようになり、
互いに干渉しなくなった。

検証: `npx tsc --noEmit` / `npx eslint .`(既知の2件のみ) / `npm run build`
すべて通過。ChromiumでバインドDOMのzIndexを実測し、裏表紙=0→先頭カード=1
→以降のカード=2,3,4,5,6,7(リストの下にあったカードほど高い値)→表表紙=9、
という意図通りの値になっていることを確認。スタック完了直前のスクリーン
ショットでも、リスト末尾だったカードが実際に一番手前(最上位)に表示されて
いることを視覚的に確認した。**WebKit実機での最終確認は引き続き未実施。**

## 7.26 ★長押しドラッグ、実機相当のマウス操作で再現する「固まる」不具合の
真因を特定して修正・入場アニメーションの視認性改善・ストックのaction
アイコン化とバッジ右上統一(2026-07-12続報)

§7.25のオートスクロール対応後もユーザーからは「やはり画面が追従しない」
「アニメーションが無い」という報告が続いた。前回の検証はPlaywrightの
`element.dispatchEvent(new PointerEvent(...))`(完全に合成されたイベント)
に依存しており、これはJSのイベントリスナーは起動するがブラウザの
「信頼できる入力(trusted event)」由来の内部状態(ネイティブスクロール判定・
pointer captureの生死など)までは再現しないという限界があった。今回は
`page.mouse.down/move/up`(CDP経由の実際のOS相当の入力)で再現テストを
やり直し、実機によりだった挙動の差を突き止めた。

**真因**: 短い移動(2〜3スロット分)では成功するが、長く速いドラッグ
(4スロット以上、複数回のswapを伴う)だけで確実に「ドラッグが二度と
終了しない」不具合が再現した。`hasPointerCapture()`で経過を実測すると、
`beginDrag`直後は確かに`setPointerCapture`が効いているのに、swap
(`setOrder`)を経るたびにキャプチャが失われていくことが判明した。
原因は、swapのたびにReactが並べ替え後の新しいDOM順序に合わせてスロット
のdiv要素を並べ替える(insertBeforeで移動する)ことで、ブラウザの実装上
これが一瞬要素をドキュメントから外す形になり、仕様通りpointer capture
が自動解除されていたこと。capture が外れた後のpointermove/pointerupは、
指が物理的に乗っている別の要素(棚の外の無関係なコンテナなど)へ流れて
しまい、どのスロットの`onPointerUp`にも届かなくなるため、`endDrag()`が
一度も呼ばれずドラッグ状態(とオートスクロールのrAFループ)が永久に
残り続けていた。ユーザー視点では「オートスクロールが効かない」と
見えるが、実際は「そもそもドラッグが正常に終了できず、内部的に暴走した
状態のまま」というより深刻な不具合だった。

**対応**: 要素側の`setPointerCapture`に依存する設計をやめ、
`pointerdown`の瞬間に`window`へ直接`pointermove`/`pointerup`/
`pointercancel`リスナーを登録し、`pointerId`の一致だけで判定する方式へ
作り替えた(`components/Binder.tsx`の`handleSlotPointerDown`)。`window`
という不変の対象にリスナーを貼るため、DOM上でスロットがどう並べ替え
られようと影響を受けず、確実に最後まで追従できる。ドラッグ確定時に
`pressRef`を明示的にnullへリセットする防御も併せて入れた。

検証: `page.mouse`による実際のドラッグ(300px、複数回のswapを伴う)を
再現し、修正前は`hasPointerCapture`が早い段階で`false`になり最終的に
`pointerup`がどのスロットにも届かず状態が固まる(shelfOrderが更新
されない)ことを確認、修正後は同じ操作で正常に`endDrag`が呼ばれ
`shelfOrder`が更新されることを確認した。短い移動での通常タップ・
素早いスワイプでの誤爆なしも再確認済み。

**その他2点の追加対応**:
- **入場アニメーションの視認性**: §7.25で`binder-in`をtranslateX方向に
  変更した際の移動量(-16px)が小さすぎ、実機で「アニメーションが無い」と
  見えるほど視認できなかった。`app/globals.css`で-64pxへ大幅に増やした。
- **ストックのactionボタンをアイコン化・バッジを右上に統一**: 「行った」
  「観た」等のactionボタンを、＋トグルと同じ20px円のチェックアイコン
  ボタンへ変更。あわせて、これまで左上に独立配置していたKEEP/WISHバッジ
  を含め、バッジ・action・プラン選択トグル・良かった(星)の4要素を
  すべて右上に1本の横並びflexでまとめて詰める構成にした(`components/
  common.tsx`のPosterCard)。バッジが右上へ移ったことで、従来top:8,
  right:8に独立配置していた良かった(星)ボタンと衝突するようになった
  ため、これも同じ列に合流させて衝突を回避した。

検証: `npx tsc --noEmit` / `npx eslint .`(既知の2件のみ) / `npm run build`
すべて通過。Chromiumでストックタブの通常カード(バッジのみ)・オーバーレイ
(バッジ+action+トグル)の両方でスクリーンショットを撮り、右上に自然に
詰まって重ならないことを確認した。**WebKit実機での最終確認は引き続き
必要だが、今回はネイティブ入力に近いCDPマウス操作で再現・検証している
ため、前回までより確度は高い。**

## 7.27 ★オートスクロールが追いつかない不具合の再修正・入場アニメーション
後の位置ズレ修正・BottomSheetのブラー誤タップ修正・モノ/バショ/タイケンの
デザイン再修正(2026-07-12続報)

§7.26でドラッグが「固まる」不具合(pointer capture消失)は直ったが、
ユーザーからは「オートスクロールで追従しない、バインダーが画面外に
行ってしまう」症状が引き続き報告された。

**オートスクロールの真因(2つ目)**: 従来の実装は、つまんだカードの
「あるべき画面位置」を`virtualIndex`(swapで離散的に飛ぶスロットindexと
指の生の移動量から逆算した連続値)と`centerRef`(コンベア全体の回転角の
基準)から間接的に計算し、そこから算出した位置がコンテナ端に近いかで
オートスクロールの要否を判定していた。この間接的な計算はgapのクランプ
処理(隣接カードの見た目上のシフト量を頭打ちにする処理)と噛み合わず、
実際の指の位置とズレることがあった。さらに致命的だったのが速度: 固定
7px/フレーム(≒420px/秒)は、現実的な速さの指の動きに全く追いつけず、
指を素早く動かすとその分だけカードがoverflow:hiddenの外へ出て見失われて
いた。
**対応**: 判定を、指の生のスクリーン座標(`pointerClientXRef`、pointermove
のたびに更新)とスクロールコンテナ自身の`getBoundingClientRect()`を
直接比較する方式に変更(間接計算を排除)。速度も固定値をやめ、端の
しきい値(72px)にどれだけ深く食い込んでいるかに応じて6px/フレームから
最大30px/フレームまで加速するようにした。Chromiumで実機のスワイプに
近いテンポ(15ms間隔で連続移動)を再現し、scrollLeftが途切れず追従し
続けること、掴んだカードがスクリーンショット上で常に可視領域内に
留まり続けることを確認した。

**入場アニメーション完了後に位置がズレる不具合**: `BinderCoverflowRow`は
初回描画の瞬間、スクロールコンテナの実測前で`containerWidth=0`のため
両端の余白(`sidePad`)も0になる。この状態のままカード(と入場アニメーション
`binder-in`)が描画され始め、直後にResizeObserverが実測して`sidePad`が
正しい値へ変わると、全カードの位置が0→正しい値へガクッと動く。以前から
存在した構造だが、`binder-in`の移動量が16pxと小さかった間は目立たず、
§7.26で64pxへ拡大したことで「入場アニメーションが終わった後に勝手に
スライドする」という体感になって顕在化した。
**対応**: `containerWidth`が実測される(0より大きくなる)まではカードを
一切描画しないようにした。これにより、カードが初めて画面に現れる時点で
既に正しい`sidePad`が確定しており、アニメーション完了後に位置が
補正される余地が構造的に無くなる。Chromiumでタブ切り替え直後から
80ms間隔でカードのbounding boxを15回(計1.12秒)計測し、2フレーム目
以降は完全に同じ座標のまま微動だにしないことを数値で確認した。

**ストックタブ: BottomSheetのブラーをタップで閉じる際、下のボタンへ
誤タップが突き抜ける不具合**: `pointerdown`で`requestClose()`を呼び
背景のフェードアウトを開始してから実際に`pointerEvents:none`へ切り替わる
までの間に、ブラウザが標準で送るpointerup後の互換`click`イベントが
割り込むと、その時点でこのブラーは`pointerEvents`が切り替わり中のため、
`click`のヒットテストがブラーの下の要素(ボタン等)まで突き抜けてしまい、
「ブラーを閉じたつもりが下のボタンを誤タップした」ことになっていた。
`closeIfSelf`/`closeOnSelfClick`(`components/BottomSheet.tsx`)の
`pointerdown`ハンドラで`e.preventDefault()`を呼び、この一連の操作に
続く互換マウスイベント(click含む)をブラウザに一切合成させないように
した。Chromiumで、シートを開いた状態でナビバー付近(ブラーの下に実際の
ボタンがある位置)をタップし、下のボタンのclickイベントが発火しないことを
イベントリスナーで確認した。

**モノ・バショ・タイケンのデザイン再修正(3件)**:
1. **モノ**: 「白地に、色のついた円を組み合わせて」という指定を受け、
   `StampMotif`を全面べた塗り(円1つ→三角1つと変遷)から、白地(PAPER)+
   同じ基準色をshade()で明度違いにした円3つを重ねる構図に作り直した。
   `BinderCoverFace`の`stamp`分岐も背景をPAPERへ、文字色もINK
   (白地なので他の白地系kindと同じデフォルト)へ変更。
2. **バショ**: 背表紙(`BinderSpineFace`)の`geo`固有分岐(45度の縞模様)を
   撤去し、単色+左右の陰影オーバーレイという他kindと共通のフォール
   バックへ統一した。
3. **タイケン**: `SideMotif`が図形を1つだけ帯の下端に置く構成だったのを、
   同じ図形をSIDE_MOTIF_COUNT(4)回、帯の全高へ`justifyContent:
   space-between`で均等配置する構成に作り直した(下地=帯の色自体は
   変更なし)。

検証: `npx tsc --noEmit` / `npx eslint .`(既知の2件のみ) / `npm run build`
すべて通過。Chromiumで(1)高速な連続ドラッグでオートスクロールが途切れず
追従することをscrollLeftの推移とスクリーンショットで確認、(2)タブ切り替え
直後からのbounding box計測でアニメーション後の位置ズレが消えたことを
数値で確認、(3)ブラー誤タップが起きないことをイベントリスナーで確認、
(4)モノ(白地+円3つ)・バショ(単色背表紙)・タイケン(4個の縦グリッド)の
見た目をスクリーンショットで確認した。**WebKit実機での最終確認は
引き続き必要。**

## 7.28 ★アーカイブのドラッグ並べ替え、ようやく真因(scroll-snap-type)を特定・
マップ縮小のスクロール連動化・ゴールバインダー再タップバグ・その他5点
(2026-07-12続報)

§7.27までの対応(pointer capture消失の修正、オートスクロールの速度・判定
改善)にもかかわらず、ユーザーからは「まだ全然改善されていないです、
というか何も変わっていないようです」という最も強い調子の再報告があった。
今回は「ドラッグして指を右に持っていったときに、右にスワイプしている
ような感じで他のバインダーが動いて欲しい」という具体的な要望とともに、
他8件の要望・バグ報告も同時に届いた。

**★ドラッグ並べ替えの真因、ついに特定**: これまで「エッジ検出方式の
オートスクロール」を何度も微調整してきたが、そもそもの設計を疑い、
つまんだカード「以外」も含めてコンベア全体を指の移動量に完全1:1で
追従させる方式(`el.scrollLeft = startScrollLeft + dragOffsetPx`、
`components/Binder.tsx`の`dragPollFrame`)に作り替えた。これは
「他のバインダーもスワイプのように動いてほしい」という要望にも直接
応える設計変更のはずだったが、実装直後の検証(`page.mouse`による連続
ドラッグ)で、scrollLeftの実測値が`[0,46,92,184,230,276,322,276]`という
非単調な(指を一方向に動かし続けているのに値が減る瞬間がある)奇妙な
推移を示した。詳細なフレーム単位のデバッグログ(`dragPollFrame`に一時的な
計測コードを仕込んで`window.__dragDebug`へ記録)を仕込んで再検証した
ところ、コード側が計算した目標値(例: offset=80ならscrollLeft=80)と、
実際にDOM上で読み取れるscrollLeftの値(92や46)が一致しないことが判明した。
**原因は`beginDrag`が`overflowX`と`touchAction`は`hidden`/`none`へ
切り替えていたが、同じ行の`scrollSnapType: "x mandatory"`(コンベアの
慣性スクロールを各アイテムの境界にピタッと止めるためのCSS)を無効化
していなかったこと。** ドラッグ中に`el.scrollLeft`へ生のpx値を代入
しても、`scroll-snap-type`がまだ有効なままだと、ブラウザのスナップ
機構がそれを「ユーザーがスクロールし始めた」とみなし、割り当てた値を
最寄りのアイテム境界(約46pxおき)へ勝手に補正してしまう。指の動きに
対してスクロール量が段刻みに量子化され、しかも中間点をまたぐタイミング
次第では前後の境界の間を行ったり来たりする(=値が減って見える)ことも
あった。これが、これまで何度エッジ検出のアルゴリズムを調整しても
「他のバインダーが追従して見えない」「掴んだバインダーが行方不明になる」
という報告が消えなかった、真の根本原因だったと考えられる(過去のセッション
はいずれも症状の"間接的な現れ"であるオートスクロールの速度・判定式ばかり
疑っており、そもそもscrollLeftへの代入自体がブラウザに横取りされていた
という、より下のレイヤーの問題には気づけていなかった)。
**対応**: `beginDrag`で`el.style.scrollSnapType = "none"`を追加し、
`endDrag`で`"x mandatory"`へ復元する(`overflowX`/`touchAction`と全く
同じ、ドラッグ中だけの一時無効化パターン)。修正後に同じデバッグログで
再検証すると、`scrollLeft`は`[0,40,80,120,160,200,240,280,320]`と、
指のx移動量(40pxずつ)に対して完全に1px単位で一致する、非単調性の
無い滑らかな推移になったことを確認した。あわせてスクリーンショットでも、
掴んだカードの左右の他のバインダーが指の動きに合わせて一緒にスライド
していく(コンベア全体が同期して動く)ことを確認した。通常の短いスワイプ
(450ms未満でキャンセルされる長押し判定)・通常タップでのフォルダ開閉、
いずれも回帰していないことも再確認済み。**この修正はscrollLeftの
実測値という数値的な証拠に基づいており、これまでのどの回よりも
根本原因への確度は高いと考えているが、繰り返し「直っていない」と
言われてきた経緯を踏まえ、実機での最終確認が必須であることは
変わらない。**

**マップ縮小をスクロール連動の連続値へ**: 従来はIntersectionObserverの
二値(枠に張り付いた/いない)にCSSトランジション(0.32s)を付けて縮小して
おり、「アニメーションではなく、スクロールするほどだんだん小さくなる
感じにしてほしい」という指定を受けた。`MapCanvas`の`stuck: boolean`
propを`shrink: number`(0=フル、1=最小72%)へ変更し、CSSトランジションを
撤去。センチネル要素の位置をスクロールイベント(`root.addEventListener
("scroll", ...)`、rAFで間引き)のたびに実測し、スクロール量に比例した
連続値をそのまま幅へ反映するようにした(`components/tabs/
ExecuteTab.tsx`)。Chromiumでスクロール位置を40pxずつ動かしながら地図幅を
計測し、`358→358→358→345→314→284→258...`と滑らかに(段階的なジャンプ
無しで)縮み、72%(258px)で正しく頭打ちになることを数値で確認した。

**ゴールバインダー、閉じた直後に再タップすると無反応になるバグ**:
バインダーを閉じた直後(`BottomSheet`の220msのフェードアウト演出の
途中)に同じバインダーを再タップすると、`openGoalId`は実際にはまだ
そのidのまま(nullになるのは閉じるアニメーション終了後)のため、
同じ値を`setOpenGoalId(id)`してもReactは「変化なし」とみなして
再レンダーせず、シートが開き直らないまま数百ms後に元の閉じる処理
だけが完了して終わっていた。2箇所を修正: (1)`GoalsTab.tsx`に
`openGoalCard`ヘルパーを新設し、既に同じidが指定されている場合は
一度`null`を経由させてから`requestAnimationFrame`で入れ直すことで、
値としては異なる更新を強制する。(2)`BottomSheet.tsx`の`requestClose`が
仕込む`setTimeout(onClose, 220)`がアンマウント後も生き残って古い
`onClose`を呼んでしまわないよう、`useEffect`のクリーンアップで
明示的に`clearTimeout`するようにした。「タップ→閉じる→150ms待つ
(まだ220msのフェードアウト中)→同じカードを再タップ」という実際の
操作テンポを再現するテストで、修正前は入力欄が見つからない(無反応)、
修正後は正常に開き直る(入力欄が見つかる)ことを確認した。

**アーカイブの入場アニメーション後にバインダーが動く不具合、安全網を追加**:
§7.27で「containerWidthが実測されるまでカードを描画しない」対策を
入れたが、ユーザーからは実機で依然としてズレが残ると報告された。
実機のみで起きるタイミング差(フォント読み込み完了に伴う再レイアウトなど、
この環境のChromiumでは再現できない要因を含む可能性がある)のため、
原因を1箇所に絞り切れないまま、対症療法にはなるが、入場アニメーションが
確実に終わり切ったタイミング(最後のカードの遅延+持続時間より少し
長めの600ms後)でもう一度`scrollLeft`と`centerRef`を0へ強制的に揃え直す
安全網を追加した(`BinderCoverflowRow`)。ユーザーがその間に実際に
スワイプ済みの場合は上書きしないよう、ドラッグ中でないこと・実際に
ズレていること(`scrollLeft`が0でないこと)を条件にしている。

**アーカイブの行が画面端で見切れる不具合**: `BinderCoverflowRow`の
スクロールコンテナが、祖先(`data-tab-scroll-root`)の16pxの左右
パディングをそのまま受け継いでいたため、両端のバインダーが画面端まで
届かず途中で見切れていた。行の根本divに`margin: "0 -16px"`を追加し、
祖先のパディング分だけ外側へ滲み出させることで、画面幅いっぱいまで
表示されるようにした。Chromiumで行の`getBoundingClientRect()`を計測し、
`{left:0, right:390}`とビューポート幅(390px)に正確に一致することを
確認した。

**タイケン(ライブ)の円グリッドが下端で見切れる不具合**: 4つの図形を
縦に積む`SideSquareCell`が`width:"100%", aspectRatio:"1/1"`でサイズを
決めていたため、セル1個の一辺が帯の"幅"基準になり、4個分の合計高さが
帯の実際の高さを超えて下端の1個が`overflow:hidden`で切れていた
(幅基準:約58.5px×4個=234px vs 実際の帯の高さ:約229px)。
`height: "${100/SIDE_MOTIF_COUNT}%", aspectRatio:"1/1"`という
高さ基準のサイズ指定に変更し、N個のセルが必ず帯の高さぴったりに
収まるようにした。Chromiumのスクリーンショットで、ライブ・コンサートの
4つの円すべてがカード内に完全に収まることを確認した。

**タイケン(グルメ)を半円連続デザインへ**: 従来は隅の四半円
(`quarterBR`)だったが、同じ「隅の四半円」系統の習い事(activity=
`quarterTL`)と、小さいグリッドの中では見分けにくいという指摘を
受けた。`EXPERIENCE_ACCENT.food`の図形を`semicircleUp`へ変更。
上記の高さ基準サイジングによりセル同士が隙間なく接するため、
半円が縦に連なった構図が自然に生まれる。

**モノをチェックマークデザインへ**: §7.27で「白地+円3つ」に作り直した
`StampMotif`を、「シンプルで太めのチェックのような」という指定を
受けてさらに作り直した。円・四半円・半円・三角・長方形という図形
ボキャブラリーには「チェック」そのものが無いため、正方形の縁
(`border-left`+`border-bottom`)だけを描き、45度回転させるという
古典的なCSSチェックマーク技法で組んだ(面は塗らず、太い線(7px)の
L字だけが見える)。回転角は45度なので「0/45/90/180度のみ」という
角度ルールの範囲内。ゴール(TargetMotif、円系)・モノの旧デザイン(円系)
との類似を避けるため、図形の系統自体を角(かど)ベースへ変えている。

検証: `npx tsc --noEmit` / `npx eslint .`(既知の2件のみ、加えてこの
セッション中に出た`react-hooks/exhaustive-deps`の不要disable警告も
削除して解消) / `npm run build`すべて通過。Chromiumで(1)ドラッグの
scrollLeft推移が完全に1:1で単調増加すること、掴んだカードと周囲の

スクロール連動の滑らかな推移、(3)ゴールの高速再タップで開き直る
こと、(4)アーカイブ行が画面端まで届くこと、(5)ライブの4円が完全に
収まること、(6)モノのチェックマークの見た目、をそれぞれ数値または
