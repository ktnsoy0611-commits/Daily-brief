# デイリーブリーフ — システム設計書（2026-07-13確定）

UI完成後の計画見直しセッションで、初期要件定義書(`IMPLEMENTATION HANDOFF.md`)を
コンセプトから読み直し、現行実装(`HANDOFF-CURRENT.md` §3〜§4が正)との照合と、
システム全体のエンドツーエンドの成立性検証を行った結果をまとめた文書。
技術面の確定事項(スキーマ分割・認証・地図方式)は `HANDOFF-CURRENT.md` §8、
本書はその上位にあるコンセプト・編集方針・プロンプト・情報源・検証ステップを扱う。

---

## 1. コンセプトとループの再定義

原典(初期要件定義書 §1):
> 趣味嗜好を貯蓄・トラッキングし、週末と日々の生活を豊かにする提案を行う

UI磨き込み期を経て、ループの意味は次のように進化した(特にバインドの意味の
確定=HANDOFF-CURRENT.md §7.8 が大きい):

1. **願いを書く**(ウィッシュ=受信箱、4ドメイン付き自由文) /
   **興味が行動から自動検出される**(週次)
2. **ブリーフ(朝夕2回)が願い・興味に応える提案を運ぶ** → スワイプでKEEP/SKIP
3. KEEPが**ストック**(4ドメインの棚+地図のピン)に貯まる
4. 出かける日に地図で選ぶ → 実行 → **バインド=実行済みの確定**
   (計画の登録ではなく、実行の記録。プランタブは「あとでやるリスト」ではない)
5. **アーカイブ**に実行済みが積層、ウィッシュに自動チェック、興味が更新され、
   次号のブリーフが良くなる(ループ)

### 成功指標

このアプリは「貯めるアプリ」ではなく「実行させるアプリ」。指標は3つ:

| 指標 | 意味 | 計測方法 |
|---|---|---|
| KEEP率 | 提案の質(興味との一致) | briefs.decisionsから集計 |
| 週あたりバインド率 | **実行**(アプリの存在意義) | bindLogから集計 |
| ウィッシュ成就リードタイム | 願いが形になる速さ | wish.addedAt→派生Itemのバインド日 |

ストックが貯まる一方でバインドが発生しない状態は、提案の質がどれだけ
高くても**失敗**と定義する。

## 2. エンドツーエンド・ウォークスルー(成立性の検証)

システムの流れを段階ごとに追い、各段の成立条件と発見したギャップを記す。

### 段階0: コールドスタート(空の状態)

興味ゼロ・KEEP履歴ゼロの初号は何を出すか。
- **フォールバック編成**: 固定ソースの新着 + ウィッシュ応答 + currentFocus。
  生成プロンプトにこの分岐を明示的に定義する(興味タグが無い場合は
  固定ソース比率を上げる)。
- 初回体験は「ウィッシュを数件書く+情報源を1〜2登録する」で成立する。
  オンボーディング画面は作らない(設定画面を作らない思想と整合)。

### 段階1: 情報が入る(sources → 巡回 → content_cache)

- 情報源は3階級: **固定**(淘汰なし)/**ユーザー登録**(淘汰なし)/**自動発掘**(淘汰対象)。
- 成立条件と対策:
  - **重複排除**: 同一イベントが複数ソースから届く。抽出時にURL正規化+
    タイトル類似で束ねる。
  - **会期の抽出**: 新着抽出プロンプトで date_start/date_end を必須で取り、
    ブリーフカードの expiresAt へ繋ぐ(→段階2のギャップ参照)。
  - **規約リスク**: RSSを優先し、HTMLスクレイピングは robots.txt と利用規約を
    ソースごとに個別確認してから固定リスト入りさせる。

### 段階2: ブリーフ生成(選定・執筆)

生成への入力(すべて実在データ。ここに無いものは書けない=創作禁止の担保):
interests(重み付き) / currentFocus / wishes(id付き・stock状態のみ) /
直近のKEEP・SKIP履歴 / 星付きdoneアイテム / **既存ストック・アーカイブの
タイトル一覧(重複抑止用)** / content_cacheの候補 / 日付・季節 / (Groundingで
取得できる範囲の天気)。

★このウォークスルーで発見したギャップ:
1. **`BriefCard.expiresAt`が存在しない**。展覧会・ライブをKEEPしても会期末が
   Itemへ引き継がれず、自動失効(`isExpiredItem`)が機能しない。生成スキーマと
   KEEP処理(BriefTab)の両方に追加する。
2. **重複抑止が未定義**。既にストック・アーカイブにあるものを再提案しない
   ため、既存タイトル一覧をプロンプトへ渡し、生成後にも照合する。
   **夕刊は同日朝刊のカードとの重複も除外する**(朝夕は同一編成のため、
   夕刊の価値=鮮度と朝刊に出せなかった候補)。
3. **「ロケーション」トリガーの再定義**。ダミーCARDSの「明日の予定の途中に」は
   カレンダー連携なしでは創作になる。**「ストックのピン近接」**(あなたのストックに
   蔵前のピンが3つある→この提案も蔵前、という自データで根拠づけられる文脈)へ
   再定義する。
4. **場所カードの座標付与は生成パイプライン内で行う**。Places API(New)の
   照会はサーバー側(キー保持)で生成時に実行し、placeId/lat/lng/写真参照を
   カードに埋めてから保存する。KEEP時のクライアント照会はしない。

### 段階3: KEEP → ストック → 地図

実座標(placeId/lat/lng)がItemに入れば、既存UI(4ドメイン棚・スタイライズド
地図への正規化描画・近接クラスタリングの「今週のおすすめ」)がそのまま機能する。
追加の設計は不要(HANDOFF-CURRENT.md §8.3で特定済みのフィールド追加のみ)。

### 段階4: 選ぶ → 実行 → バインド → アーカイブ

完全にクライアント内で完結し、実装済み。
- `pendingReview`(「行きましたか？」)は新しいバインド意味論の下でも整合する:
  選んだが綴じずに日を跨いだ=実行したか不明、という状態の受け皿。
- 事後評価は「良かった」星のみ。感想メモ等の入力は**足さない**
  (タスク化しない・記録を義務にしない思想を維持)。星は生成入力(正の嗜好
  シグナル)として使う。

### 段階5: 学習ループ

- **興味検出のGemini化**(週1): wishes+itemタイトル+星から重み付きタグを生成。
  既存タグとのマージは「急変させない」方針(新規タグは低い重みで参入、
  出現しなくなったタグは減衰で退場。1回の週次実行で総入れ替えしない)。
- **ソーススコア**: KEEP率で加点・旗で減点 → 週次で自動発掘ソースの下位を
  入れ替え(多腕バンディット)。前提となるカード→ソースの`sourceId`配線は
  HANDOFF-CURRENT.md §8.3で特定済み。
- セレンディピティ枠の採択率も記録し、将来の枚数自動調整の材料にする。

### 横断: 失敗時の体験

- 生成失敗・未生成 → 「休刊」表示+再試行ボタン。前号は残さない
  (鮮度こそが価値のため、古い号を今日の顔にしない)。
- API呼び出しカウンタの閾値超過時も同じ「休刊」に落ちる(課金安全装置)。

## 3. プロンプト設計(6種)

### 3-1. ブリーフ生成(Flash + Grounding、1日2回)

- **ペルソナ**: 「あなたは私専属の編集者。私の願いと興味を知り尽くし、
  検索上位の浅い情報ではなく、自分では辿り着けないニッチで質の高い情報を
  1日2回、雑誌の号のように届ける」
- **大原則**(システムプロンプトに明記):
  1. 事実の創作禁止。検索・content_cacheで確認できた事実だけで書く
  2. 動線・予定・貯金・運動への言及禁止(そのデータは存在しない)
  3. 全カードに参照元URL必須
  4. 渡した既存一覧(ストック・アーカイブ・同日朝刊)との重複禁止
- **編成(初期値・バランス型)**: 10枚 = ウィッシュ応答2 + 興味一致4 +
  タイムリー2 + 登録サイト1 + セレンディピティ1。
  コールドスタート時(興味タグなし)は 固定ソース新着を厚くする。
  ※この比率はKEEP率・旗の実績で自動調整する前提の初期値。
- **出力スキーマ**: BriefCard準拠のJSON。必須: title/body/kind/trigger/
  sourceUrl/sourceId。任意: area/**expiresAt**/meta/price/sourceWishId(ウィッシュ
  応答時はidで明示)/serendipity。
- **トーン**: 現行ダミーCARDSの2〜3枚をfew-shotとして同梱(簡潔・具体的・
  「なぜ今か」が一行で分かる)。

### 3-2. 新着抽出(Flash-Lite、毎日30〜50回)

「このHTML/RSSから新着のイベント・記事を抽出せよ。出力:
`{title, url, date_start?, date_end?, area?, summary}`の配列。
ページに書かれていない情報は出力しない」。検索ではなく抽出なので極めて安価。

### 3-3. 興味検出(Flash-Lite、週1)

wishes+itemタイトル+星付きdoneを入力に、重み付き興味タグを出力。
既存タグ一覧も渡し「維持/重み調整/新規/減衰」の差分形式で返させる
(総入れ替えを構造的に不可能にする)。

### 3-4. ソース発掘(Flash + Grounding、週1)

興味タグごとに「質の高い情報を継続的に発信している個人ブログ・地域専門
メディア・専門サイトを探せ。大手まとめサイト・アグリゲータは除外」。
出力: url/label/なぜ良いか。origin='auto'でプールに追加。

### 3-5. URL解析(Flash-Lite、手動追加時のみ)

任意ページから場所/イベント情報を抽出(`mockParseUrl`の置き換え)。
GoogleマップURLはGeminiを使わずPlaces APIへ直行。

### 3-6. セレンディピティ(ブリーフ生成に内包)

①興味の隣接キーワードを発想(ここだけ創作OK・事実不要)→②そのキーワードで
Grounding検索→③実在の情報に一言添えて1枚だけ入れる。実在しない体験を
書くことは一度もない。

## 4. 情報源戦略(生活圏=東京23区)

固定ソースの**カテゴリ**(具体URLはフェーズC実装時にRSS有無・robots・規約を
個別検証してから確定する。ここに未検証のURLリストは書かない):

| カテゴリ | 拾うもの | 更新頻度の想定 |
|---|---|---|
| 展覧会一覧系メディア | 美術館・ギャラリーの会期情報(expiresAtの主要供給源) | 毎日 |
| ミニシアター上映スケジュール | 単館系の上映作品・最終週情報 | 毎日 |
| ライブハウス・ホール公式 | 公演告知(発表直後の検知が価値) | 毎日 |
| 映画公開スケジュール | 公開日・上映館 | 週次で十分 |
| 主要美術館の公式ニュース | 一次情報での裏取り・詳細 | 週次 |

- **ユーザー登録**(設定画面の既存UI): rateyourmusic等。ランキング系は
  上位N件からのランダムサンプリングで「ある程度ランダムに」を実現
  (初期定義書§5の方針を維持)。
- **自動発掘**(週次): §3-4のプロンプトで補充し、KEEP率・旗のスコアで
  下位を入れ替える。固定・ユーザー登録は淘汰しない。

## 5. 機能させるためのステップ

フェーズA(基盤)・フェーズB(URL解析・実座標)は HANDOFF-CURRENT.md §8.4 の
とおり。本見直しでの追加・変更:

- **フェーズC-0(新設): プロンプト実験場**。UIに配線する前に、実データの
  スナップショット(自分のwishes/interests/ストック)に対してブリーフ生成を
  単体実行し、デッキJSONの品質を反復評価する開発スクリプトを作る。
  - **品質バー**: 10枚中、参照URLが実際に開けて内容が一致するもの≥7 /
    自分にとって本当に新しい情報≥3 / 既存ストックとの重複0 / 事実の創作0。
  - このバーを超えるまでプロンプトを磨いてから本配線する。
    「生成はできたが質が低い」状態でUIに流すと、KEEP率が下がり学習ループ
    全体が汚染されるため、ここが品質の関所になる。
- **フェーズC(本配線)**: 生成パイプライン + **PWAプッシュ通知**(朝刊・夕刊の
  到着通知。iOS 16.4+のWeb Push、ホーム画面追加済みPWAが前提)。
- **フェーズD**: 興味検出のGemini化(週1)、ソース淘汰の本稼働、
  セレンディピティ調整、ライブ告知の毎日チェック、カレンダー連携ほか。
